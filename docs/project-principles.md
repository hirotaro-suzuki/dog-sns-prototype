# プロジェクト原則

最終更新: 2026-07-05

この文書は、店舗分散型SNS自動投稿システムを進めるうえで、スレッドが変わってもぶれないための最上位方針である。

細かな画面仕様やDB仕様は `dog_sns_design.md` を正とするが、判断に迷った場合は、この文書の原則を優先する。

## 0. 作業開始直後の最優先ルール

Codexは、新しいスレッドやコンテキスト再開直後に、まずGitHub連携ツールでGitHub `main` 上の指定文書を読む。

以下は、確認目的であっても先に行わない。

- Dropbox配下を探索する
- ローカルPC上のフォルダを探索する
- 手元フォルダや古い複製リポジトリを確認する
- ローカルへcloneする
- ローカルの `.git` や作業コピーを正本の代わりに見る
- GitHub連携で詰まった時に、説明なしでローカルやDropboxへ回避する

「正本にしない」とは、判断材料にしないだけでなく、作業開始時の確認経路としても使わない、という意味である。

GitHub連携ツールで読めない、更新できない、Vercel確認ができないなどの問題が起きた場合は、ローカルへ逃げず、その時点で止まり、何ができて何ができないかをユーザーへ説明する。

過去に、ユーザーがGitHubへログインし直すことでGitHub連携が復旧したことがある。そのため、GitHub連携ツール、GitHub上の確認、Vercel status確認、Vercelログ確認が認証や接続状態で失敗した場合、CodexはローカルやDropboxへ回避する前に、ユーザーへGitHubまたはVercelへのログイン状態確認を依頼する。ユーザーがログイン操作を行えば進められる可能性がある場合は、そのことを専門用語を減らして一歩ずつ案内する。

どうしても一時的な作業コピーが必要に見える場合でも、Codexは先に以下を説明し、ユーザーの明示確認を得るまで進めない。

- なぜGitHub連携だけでは足りないのか
- どこに一時作業場所を作るのか
- 何を読み、何を書き、作業後に何を削除するのか
- Dropboxや既存の手元フォルダに触れないことをどう守るのか

この章は他の作業効率より優先する。急いでいても、GitHub連携ツールより先にローカルやDropboxを見ない。

## 1. このプロジェクトの立ち位置

このプロジェクトは、ユーザー本人の事業として長期運用するものではなく、友人が経営する飲食店を支援するためのものである。

ある程度使える形になったら、店舗側または運用主体へ引き渡し、ユーザー本人は手を引く前提で進める。

そのため、機能を作ることだけでなく、次の人が理解し、引き継ぎ、運用できる状態にすることを重視する。

## 2. ユーザー本人に依存しない

ユーザー本人が、将来すべての経緯、判断、設定、操作手順を覚えている前提にしない。

仕様、判断理由、作業手順、未決事項、注意点はGitHub上の文書に残す。

将来のユーザー本人、次のCodexスレッド、別の作業者、店舗側担当者が読んでも再開できる状態を保つ。

## 3. クラウドを正とする

このプロジェクトでは、GitHub、Vercel、Supabaseを正とする。

- ソースコードと文書は GitHub `main` を正本とする
- 実行、ビルド、公開確認は Vercel を正とする
- DB、Storage、Auth、店舗設定、担当者、枠、完成画像は Supabase を正とする
- ローカルPC、Dropbox、個人フォルダ、複製リポジトリを正本にしない
- Codexは、GitHub上の確認・更新をまずGitHub連携ツールで行う
- GitHub連携ツールで更新できない場合は、ローカルやDropboxへ退避せず、そこで止まって状況をユーザーへ説明する
- GitHub mainへ反映した後は、必ずVercelの対象コミットのステータスを確認する。`pending` なら待ち、`failure` ならVercelログを読み、原因を直して再度 `success` まで確認する
- GitHub反映、Vercelビルド成功、Vercel上の実画面確認は別物として扱い、どれを確認したかを分けて報告する
- ローカル作業コピー、clone、Dropbox配下の探索が必要に見える場合でも、先に理由と影響範囲を説明し、ユーザー確認なしに進めない

困った時にローカルへ退避するのではなく、クラウド上で引き継げる形を守る。

## 4. Codex一本で進める

このプロジェクトでは、Codexを中心に進める。

以前のように、複数AIや複数ツールの間で、ユーザーが大量のコピー＆ペーストをして情報を運ぶ運用は避ける。

Codexは、GitHub上の文書、コード、履歴を読み、必要な確認、実装、記録、次作業整理まで一貫して担当する。

ユーザーに依頼する作業は、Codexから実行できないクラウド画面操作、実機確認、最終判断などに限定する。その場合も、専門用語を減らして一歩ずつ案内する。

## 5. 先走らない

Codexは、未確定事項を勝手に決めて実装しない。

目的、OK条件、保存先、運用者、影響範囲が曖昧な機能は、実装前に候補を出してユーザーと確認する。

速度よりも、確認、合意、引き継ぎやすさを優先する。

特に、DB構造、SNS投稿機能、AI連携、権限管理、削除機能、ロゴ・枠方針、店舗運用に関わる変更は、先に設計と確認を行う。

## 6. TDDを基本姿勢にする

このプロジェクトでは、実装前に「何ができたらOKか」を先に決める。

自動テストできる部分は自動テストで確認する。

iPad Safari、印刷、見た目、現場操作など自動化しにくい部分は、手動確認リストを先に作る。

画面や操作フローを変更する場合は、単に「ボタンがある」「表示が変わる」だけでなく、状態の変化を確認条件に含める。

例:

- ログイン前
- ログイン直後
- 再読み込み後
- ログアウト後
- もう一度ログインした後
- エラー時
- 対象データが0件の時
- 対象データがある時

Codexは、小さなUI変更でも実装前に最低限以下を整理する。

- 目的
- 変更対象
- OK条件
- 画面状態ごとの確認項目
- 自動確認できること
- VercelやiPad Safariで手動確認が必要なこと
- 今回確認できないこと

実装後は、何を確認したか、何が未確認かを作業記録に残す。

未確認のものを「できた」と報告しない。GitHubに反映しただけの場合は「GitHubには反映済み、Vercel上の実操作確認は未確認」と明確に分けて報告する。

ここでいうTDDは、プログラムのテストだけでなく、確認条件を先に決めてから進める姿勢を含む。

## 7. 疎結合を守る

現場側、本部側、将来のSNS投稿機能は、DBとStorageでつなぐが、画面や処理はできるだけ独立させる。

現場側は、店舗スタッフが撮影、編集、印刷、同意後保存を迷わず行えることを優先する。

本部側は、保存済み写真、店舗、担当者、枠、将来の投稿素材を管理できることを優先する。

将来のSNS投稿機能は、現場の撮影体験に直接絡めすぎない。

片方の変更が、もう片方を壊したり重くしたりしない構成を守る。

## 8. 店舗側UIは軽く単純にする

店舗側画面は、接客中に使うことを前提にする。

スタッフは、お客様対応をしながら、写真撮影、写真選択、担当者選択、必要最小限の加工、印刷、同意後保存を行う。

そのため、店舗側UIは複雑にしない。

- 入力項目を増やしすぎない
- 選択肢を増やしすぎない
- 画面遷移を増やしすぎない
- 現場スタッフが判断に迷う機能を増やしすぎない
- 犬情報を細かいフォームで入力させない

店側のiPadは性能や世代がまちまちであるため、ローカル処理を重くしない。

画像は早い段階で軽量化し、ブラウザ内で保持するデータ量を抑える。

## 9. 文書を育てる

作業の区切りでは、コードだけでなく文書も確認する。

変更した仕様、判断、未確認事項、次にやる候補は、必要に応じてREADME、設計書、引き継ぎ文書、作業ログへ反映する。

文書を更新しない場合でも、なぜ更新不要と判断したかを作業報告で短く残す。

## 10. Codexからの気づきを残す

各作業区切りでは、Codexは作業内容だけでなく、作業中に気づいたリスク、違和感、改善案、未決事項を1つ以上提示する。

ただし、Codexからの気づきは即実装しない。

ユーザーが確認し、必要と判断したものだけを、次の作業候補またはドキュメント更新候補にする。

Codexの力は、先走った実装ではなく、気づきの提示、整理、確認に使う。

## 11. 作業報告の形式を固定する

Codexは、作業報告の最後に以下を必ず含める。

短い作業でも省略しない。最低限、確認済み、未確認、Codexからの気づきは必ず書く。

```text
作業内容
- 今回変更、確認、整理したこと

確認済み
- GitHub上で確認したこと
- Vercel上で確認したこと
- iPad Safariや実機で確認したこと

未確認
- まだ確認できていないこと
- ユーザー確認が必要なこと
- Vercel反映待ちのこと

次に進む候補
- 今すぐ実装せず、候補として出すこと

Codexからの気づき
- 作業中に気づいたリスク、違和感、改善案、未決事項を最低1つ
- ただし、ユーザー確認なしに即実装しない

更新した文書
- 更新した文書名

更新しなかった文書
- 更新不要と判断した文書名と理由
```

未確認のものを確認済みのように書かない。
GitHub反映、Vercel反映、iPad Safari確認、実機印刷確認は分けて書く。

## 12. スレッド移行を先回りして準備する

Codexは、コンテキスト圧縮の直前通知を前提にしない。

コンテキスト圧縮は予告なく起きる可能性があるため、「圧縮されそうになったら知らせる」ではなく、作業の区切りごとに先回りして引き継ぎメモを作る。

特に以下の場合は、実装を続ける前にドキュメント更新や次スレッド移行を提案する。

- 重要な実装に入る前
- DB、Storage、Vercel、運用手順に関わる判断をした時
- チェックポイントを越えた時
- 複数ファイルにまたがる変更を行った時
- 作業が長くなり始めた時
- 話題が大きく切り替わる時
- ユーザー判断が必要な未決事項が増えた時

スレッド移行のために整理する内容:

- 今回決めたこと
- 今回変更したこと
- 確認済みのこと
- 未確認のこと
- 次に進む候補
- 次スレッドで最初に読む文書
- 次スレッドで最初に確認すべきOK条件
- GitHub main上で更新済みの文書と、更新していない文書

Codexは、作業が長くなってきた場合や、話題が大きく切り替わる場合には、ユーザーに「ここで一度引き継ぎメモを作って新スレッドに移るのが安全です」と提案する。

スレッド移行は、作業の中断ではなく、引き継ぎ品質を保つための通常運用として扱う。
