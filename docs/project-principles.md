# プロジェクト原則

最終更新: 2026-07-04

この文書は、店舗分散型SNS自動投稿システムを進めるうえで、スレッドが変わってもぶれないための最上位方針である。

細かな画面仕様やDB仕様は `dog_sns_design.md` を正とするが、判断に迷った場合は、この文書の原則を優先する。

## 1. このプロジェクトの立ち位置

このプロジェクトは、ユーザー本人の事業として長期運用するものではなく、友人が経営する飲食店を支援するためのものである。

ある程度使える形になったら、店舗側または運用主体へ引き渡し、ユーザー本人は手を引く前提で進める。

そのため、機能を作ることだけでなく、次の人が理解し、引き継ぎ、運用できる状態にすることを重視する。

## 2. ユーザー本人に依存しない

ユーザー本人が、将来すべての経緯、判断、設定、操作手順を覚えている前提にしない。

仕様、判断理由、作業手順、未決事項、注意点はGitHub上の文書に残す。

将来のユーザー本人、次のCodexスレッド、別の作業者、店舗側担当者が読んでも再開できる状態を保つ。

## 3. クラウドを正とする

このプロジェクトでは、GitHub、Vercel、Supabaseを正とする。

- ソースコードと文書は GitHub `main` を正本とする
- 実行、ビルド、公開確認は Vercel を正とする
- DB、Storage、Auth、店舗設定、担当者、枠、完成画像は Supabase を正とする
- ローカルPC、Dropbox、個人フォルダ、複製リポジトリを正本にしない

困った時にローカルへ退避するのではなく、クラウド上で引き継げる形を守る。

## 4. Codex一本で進める

このプロジェクトでは、Codexを中心に進める。

以前のように、複数AIや複数ツールの間で、ユーザーが大量のコピー＆ペーストをして情報を運ぶ運用は避ける。

Codexは、GitHub上の文書、コード、履歴を読み、必要な確認、実装、記録、次作業整理まで一貫して担当する。

ユーザーに依頼する作業は、Codexから実行できないクラウド画面操作、実機確認、最終判断などに限定する。その場合も、専門用語を減らして一歩ずつ案内する。

## 5. 先走らない

Codexは、未確定事項を勝手に決めて実装しない。

目的、OK条件、保存先、運用者、影響範囲が曖昧な機能は、実装前に候補を出してユーザーと確認する。

速度よりも、確認、合意、引き継ぎやすさを優先する。

特に、DB構造、SNS投稿機能、AI連携、権限管理、削除機能、ロゴ・枠方針、店舗運用に関わる変更は、先に設計と確認を行う。

## 6. TDDを基本姿勢にする

このプロジェクトでは、実装前に「何ができたらOKか」を先に決める。

自動テストできる部分は自動テストで確認する。

iPad Safari、印刷、見た目、現場操作など自動化しにくい部分は、手動確認リストを先に作る。

実装後は、何を確認したか、何が未確認かを作業記録に残す。

ここでいうTDDは、プログラムのテストだけでなく、確認条件を先に決めてから進める姿勢を含む。

## 7. 疎結合を守る

現場側、本部側、将来のSNS投稿機能は、DBとStorageでつなぐが、画面や処理はできるだけ独立させる。

現場側は、店舗スタッフが撮影、編集、印刷、同意後保存を迷わず行えることを優先する。

本部側は、保存済み写真、店舗、担当者、枠、将来の投稿素材を管理できることを優先する。

将来のSNS投稿機能は、現場の撮影体験に直接絡めすぎない。

片方の変更が、もう片方を壊したり重くしたりしない構成を守る。

## 8. 店舗側UIは軽く単純にする

店舗側画面は、接客中に使うことを前提にする。

スタッフは、お客様対応をしながら、写真撮影、写真選択、担当者選択、必要最小限の加工、印刷、同意後保存を行う。

そのため、店舗側UIは複雑にしない。

- 入力項目を増やしすぎない
- 選択肢を増やしすぎない
- 画面遷移を増やしすぎない
- 現場スタッフが判断に迷う機能を増やしすぎない
- 犬情報を細かいフォームで入力させない

店側のiPadは性能や世代がまちまちであるため、ローカル処理を重くしない。

画像は早い段階で軽量化し、ブラウザ内で保持するデータ量を抑える。

## 9. 文書を育てる

作業の区切りでは、コードだけでなく文書も確認する。

変更した仕様、判断、未確認事項、次にやる候補は、必要に応じてREADME、設計書、引き継ぎ文書、作業ログへ反映する。

文書を更新しない場合でも、なぜ更新不要と判断したかを作業報告で短く残す。

## 10. Codexからの気づきを残す

各作業区切りでは、Codexは作業内容だけでなく、作業中に気づいたリスク、違和感、改善案、未決事項を1つ以上提示する。

ただし、Codexからの気づきは即実装しない。

ユーザーが確認し、必要と判断したものだけを、次の作業候補またはドキュメント更新候補にする。

Codexの力は、先走った実装ではなく、気づきの提示、整理、確認に使う。

## 11. スレッド移行を早めに準備する

Codexは、コンテキスト圧縮が必要になる前に、自律的に新スレッドへの移行準備を申し出る。

長い作業、重要な設計判断、複数ファイルにまたがる変更、DBや運用に関わる変更を行った後は、次スレッドでも迷わず再開できるように、以下を整理する。

- 今回決めたこと
- 今回変更したこと
- 確認済みのこと
- 未確認のこと
- 次に進む候補
- 次スレッドで最初に読む文書
- 次スレッドで最初に確認すべきこと

Codexは、作業が長くなってきた場合や、話題が大きく切り替わる場合には、ユーザーに「ここで一度引き継ぎメモを作って新スレッドに移るのが安全です」と提案する。

スレッド移行は、作業の中断ではなく、引き継ぎ品質を保つための通常運用として扱う。
