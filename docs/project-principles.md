# プロジェクト原則


最終更新: 2026-07-05


この文書は、店舗分散型SNS自動投稿システムを進めるうえで、スレッドが変わってもぶれないための最上位方針である。


細かな画面仕様やDB仕様は `dog_sns_design.md` を正とするが、判断に迷った場合は、この文書の原則を優先する。


## 0. 作業開始直後の最優先ルール


Codexは、新しいスレッドやコンテキスト再開直後に、まずGitHub連携ツールでGitHub `main` 上の指定文書を読む。


以下は、確認目的であっても先に行わない。


- Dropbox配下を探索する
- ローカルPC上のフォルダを探索する
- 手元フォルダや古い複製リポジトリを確認する
- ローカルへcloneする
- ローカルの `.git` や作業コピーを正本の代わりに見る
- GitHub連携で詰まった時に、説明なしでローカルやDropboxへ回避する


「正本にしない」とは、判断材料にしないだけでなく、作業開始時の確認経路としても使わない、という意味である。


GitHub連携ツールで読めない、更新できない、Vercel確認ができないなどの問題が起きた場合は、ローカルへ逃げず、その時点で止まり、何ができて何ができないかをユーザーへ説明する。


過去に、ユーザーがGitHubへログインし直すことでGitHub連携が復旧したことがある。そのため、GitHub連携ツール、GitHub上の確認、Vercel status確認、Vercelログ確認が認証や接続状態で失敗した場合、CodexはローカルやDropboxへ回避する前に、ユーザーへGitHubまたはVercelへのログイン状態確認を依頼する。ユーザーがログイン操作を行えば進められる可能性がある場合は、そのことを専門用語を減らして一歩ずつ案内する。


ブラウザ上のGitHubログイン、ChatGPTアプリのGitHub接続表示、CodexのGitHub連携、Gitのpush資格情報は別物として扱う。表示上は接続済みでもCodex側が更新できない場合があるため、同じ操作を繰り返さず、どの経路が失敗しているかを切り分けて報告する。Chrome上のGitHub画面を使ったWeb UI経由の更新は通常ルートではなく、GitHub連携とGit pushが壊れている場合の最後の迂回策とし、ユーザーの明示許可を得てから行う。


どうしても一時的な作業コピーが必要に見える場合でも、Codexは先に以下を説明し、ユーザーの明示確認を得るまで進めない。


- なぜGitHub連携だけでは足りないのか
- どこに一時作業場所を作るのか
- 何を読み、何を書き、作業後に何を削除するのか
- Dropboxや既存の手元フォルダに触れないことをどう守るのか


この章は他の作業効率より優先する。急いでいても、GitHub連携ツールより先にローカルやDropboxを見ない。


## 1. このプロジェクトの立ち位置


このプロジェクトは、ユーザー本人の事業として長期運用するものではなく、友人が経営する飲食店を支援するためのものである。


ある程度使える形になったら、店舗側または運用主体へ引き渡し、ユーザー本人は手を引く前提で進める。


そのため、機能を作ることだけでなく、次の人が理解し、引き継ぎ、運用できる状態にすることを重視する。


## 2. ユーザー本人に依存しない


ユーザー本人が、将来すべての経緯、判断、設定、操作手順を覚えている前提にしない。


仕様、判断理由、作業手順、未決事項、注意点はGitHub上の文書に残す。


将来のユーザー本人、次のCodexスレッド、別の作業者、店舗側担当者が読んでも再開できる状態を保つ。


## 3. クラウドを正とする


このプロジェクトでは、GitHub、Vercel、Supabaseを正とする。


- ソースコードと文書は GitHub `main` を正本とする
- 実行、ビルド、公開確認は Vercel を正とする
