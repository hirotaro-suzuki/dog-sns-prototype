# 所有権・運用引き渡しメモ

最終更新: 2026-07-06

この文書は、将来このシステムを店舗側または運用主体へ引き渡すための準備メモです。

このプロジェクトは、ユーザー本人の長期事業ではなく、友人が経営する飲食店を支援するためのものです。ある程度使える形になったら、ユーザー本人は手を引ける状態にすることを目標にします。

最上位原則は `docs/project-principles.md`、細かな仕様は `dog_sns_design.md`、本番前の具体的な確認項目は `docs/production-readiness-checklist.md` を参照します。

## 引き渡しの基本方針

- GitHub、Vercel、Supabaseを正とする。
- ローカルPC、Dropbox、個人フォルダ、ユーザー本人の記憶に依存しない。
- 本番環境の値や秘密情報はGitHubへ書かない。
- 次の運用者が、GitHub、Vercel、Supabaseを見れば再開できる状態にする。
- 店舗側の日常操作と、本部側の管理操作を分ける。
- SNS投稿、自動投稿、Instagram連携は後続フェーズとし、引き渡し時に「自動でできる」と誤解されないようにする。

## 引き渡し対象

### GitHub

- ソースコードと文書の正本。
- `main` ブランチを最新版として扱う。
- 店舗側へ引き渡す場合は、リポジトリの所有権移管または店舗側Organizationへの移動を検討する。
- 移管前に、不要な個人情報、古いローカル前提の記述、使わない一時文書が残っていないか確認する。

### Vercel

- GitHubリポジトリから自動デプロイする実行環境。
- 本番プロジェクトは、可能であれば店舗側または運用主体のVercel Team配下へ移す。
- 環境変数はVercel側で管理し、GitHubへ値を書かない。
- `ADMIN_MAINTENANCE_PIN` などの秘密値は、引き渡し時に新しい運用者が再設定できるようにする。

### Supabase

- DB、Storage、Auth、店舗設定、担当者、枠、完成画像の正本。
- 可能であれば、店舗側または運用主体のOrganization配下へ移す。
- Service Role Keyはサーバー側だけで使い、ブラウザへ出さない。
- anon keyはブラウザで使う前提だが、RLSでアクセス範囲を制限する。
- 本番DBへ適用したSQLは、`supabase/migrations/` にも記録する。

## 環境変数

主な環境変数は以下です。

```text
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY
ADMIN_MAINTENANCE_PIN
```

`NEXT_PUBLIC_` で始まる値はブラウザに公開されます。秘密にしたい値には使いません。

## 運用担当の分担

### 店舗担当者

店舗担当者は、原則として店舗側画面だけを使います。

主な操作:

- 店舗コードとPINでログインする。
- 写真枠を選ぶ。
- わんちゃんを撮影する。
- 3枚の中から写真を選ぶ。
- 撮影担当者を選ぶ。
- 必要な場所だけ手動でモザイクする。
- 短い文字を写真上へ入れる。
- 完成画像を確認する。
- 印刷する。
- お客様からSNS掲載OKをもらった場合だけ保存する。

### 本部担当者

本部担当者は、原則として `/admin` を使います。

主な操作:

- 保存済み完成画像を一覧確認する。
- 保存済み完成画像の一言メモと本部確認状態を更新する。
- 保存済み完成画像を非表示、復帰する。
- 店舗マスタを編集する。
- 担当者マスタを追加、停止、編集する。
- 店舗フレームを登録、差し替えする。

## 削除・非表示の方針

店舗側、本部側ともに、最初は完全削除より非表示を優先します。

```text
非表示操作
→ assets.status を archived にする
→ 日常一覧には出さない
→ 本部側では確認・復帰できる
```

完全削除は、運用責任者または本部管理者だけが行う方針にします。
写真や記録を誤って消すと戻せないため、店舗側に完全削除を持たせません。

## 引き渡し前に決めること

- GitHubリポジトリの所有者と管理者。
- Vercelプロジェクトの所有Team、請求、環境変数の管理者。
- Supabaseプロジェクトの所有Organization、請求、APIキー、RLS、バックアップ方針。
- 店舗マスタ、担当者マスタ、枠画像の更新担当者。
- 店舗側の日常操作担当者。
- 本部側の管理担当者。
- 保存済み写真の非表示、完全削除、復旧のルール。
- 本部確認状態 `new`、`candidate`、`hold`、`rejected` の運用ルール。
- 緊急時にクラウド保存や本部管理画面の利用を止める手順。
- Canon SELPHY / AirPrintの実機印刷手順。

## 現時点の未整理事項

- ロゴは単独素材ではなく、写真枠のデザイン内に含める方針へ寄せる。
- 既存のロゴURL、ロゴアップロード、ロゴ管理は、後で削除または非表示にする候補。
- `public/store-frames/karuizawa-premium-*.svg` は正式ロゴ画像を使っていない仮デザインであり、社長確認用・本番用の完成版として扱わない。
- `public/store-frames/karuizawa-simple-*.svg` も問題切り分け用の単純枠であり、完成版ではない。
- `/admin/frame-cleanup` は軽井沢店の枠を整理するために追加した一時補助画面。恒久機能として残すか、後で削除するかを決める。
- 正式ロゴ入りL版フレームを、まず軽井沢店1色で完成品質まで作り、見え方を確認してから他店舗や色展開へ広げる。
- SNS投稿素材管理、投稿文作成、Instagram連携は後続フェーズとして扱う。

## この文書の更新ルール

所有権、運用分担、GitHub、Vercel、Supabase、環境変数、権限、削除方針、引き渡し手順に関わる変更をした場合は、この文書を確認します。

ただし、日付ごとの実装済み一覧や作業ログはこの文書に増やしません。実装状況は `README.md`、`dog_sns_design.md`、該当する作業ログ、または `docs/DOC_INDEX.md` から辿れる文書へ記録します。
