# 所有権・運用引き渡しメモ

最終更新: 2026-07-04

このプロジェクトは、最終的に店舗または運用主体へ引き渡す前提で管理する。
GitHub、Vercel、Supabaseを正として、特定のPC、Dropbox、個人フォルダ、ローカル複製リポジトリがなくても引き継げる状態を保つ。

## 基本方針

- GitHubの `main` をソースコードの正本にする。
- 実行、ビルド、公開確認はVercelを正とする。
- DB、Storage、店舗設定、担当者、枠、完成画像はSupabaseを正とする。
- ロゴは単独素材ではなく、写真枠のデザイン内に含める方針にする。
- 既存のロゴURL、ロゴアップロード、ロゴ管理は、後で削除する予定の古い機能として扱う。
- ローカルPCやDropboxなど、個人所有の環境を引き継ぎ前提にしない。
- 本番環境の値はコードに書かず、VercelとSupabase側で差し替えられるようにする。
- 店舗担当者は日常操作だけ、本部担当者はマスタ管理と保存画像管理を担当する。
- SNS投稿そのものは後続フェーズとし、撮影、編集、同意後保存までを疎結合に保つ。

## この文書の更新ルール

この文書は、単なるメモではなく、将来の所有権移管と運用引き渡しのための基準文書として扱う。

GitHub、Vercel、Supabase、環境変数、DB構造、Storage、管理画面、店舗画面、権限、削除方針、または引き渡し手順に関わる変更を行った場合は、作業の最後にこの文書も確認する。現状、未実装事項、注意点、チェックリストのいずれかが変わる場合は、他の設計書やREADMEと合わせてこの文書も更新する。

特に、次の変更を行った時は更新漏れを避ける。

- 店舗側または本部側の役割が変わる変更。
- Supabaseのテーブル、カラム、Storageバケット、RLS、SQL migrationを追加・変更する作業。
- Vercelの環境変数、デプロイ、管理PIN、外部連携に関わる作業。
- GitHub、Vercel、Supabaseの所有者、管理者、移管方法に関わる作業。
- 保存済み写真、枠、ロゴ、担当者、店舗マスタ、SNS投稿素材の運用ルールが変わる作業。

更新しない場合でも、作業後の報告で「この文書への変更は不要」と判断理由を短く残す。

## 2026-07-04時点で実装済み

店舗側:

- 店舗コード + PINで店舗ログインできる。
- 店舗情報、担当者一覧、枠、テーマ色をSupabaseから取得できる。
- 店舗ごとに最大3種類の有効な枠を表示し、撮影前に選択できる。
- ロゴ単体画像は店舗画面や完成画像へ重ねず、ロゴ入り枠画像を使う方針へ変更済み。
- iPad Safariで最大3枚まで撮影できる。
- 撮影画像は長辺2400pxを上限に軽量化し、ブラウザ内に一時保持する。
- 3枚から1枚を選び、その後で担当者を選択できる。
- 写真の移動、拡大縮小、回転、手動モザイク、短い文字入れができる。
- わんちゃん情報はDBに保存せず、写真上の短い文字として完成画像に焼き込む。
- 完成画像には、選択中の枠に設定された日付位置、文字サイズ、色で日付を描画できる。
- 完成画像確認画面から、印刷、保存、編集へ戻る操作ができる。
- SNS掲載OKをもらった完成画像だけをSupabase Storageと `assets` テーブルへ保存できる。

本部側 `/admin`:

- 本部PINでログインできる。PINの値はGitHubに記録しない。
- 保存済み写真の一覧、説明文更新、非表示、復帰ができる。
- 店舗マスタの表示名、SNS表示名、フレームURL、テーマカラー、有効/無効を編集できる。
- 担当者マスタの追加、名前変更、役割、有効/無効、並び順を編集できる。
- 店舗ごとの写真枠を追加、アップロード、差し替え、有効/無効、標準枠、並び順を管理できる。
- 既存の店舗ロゴ画像アップロード機能は残っているが、今後削除予定の古い管理機能として扱う。

DB・素材管理:

- `stores`、`staff_members`、`assets`、`store_frames` を中心に運用する。
- `store_frames` には、枠ごとの日付表示設定用カラムを追加済み。
- 日付表示設定カラムは、`date_enabled`、`date_x`、`date_y`、`date_font_size`、`date_color`。
- 追加SQLは `supabase/migrations/20260704_frame_date_settings.sql` に記録済み。
- `supabase/schema.sql` にも、日付表示設定カラムと制約を反映済み。
- `stores.logo_url` など既存ロゴ関連項目は、すぐには消さず、枠画像への移行後に整理する。

## 仮の運用分担

この分担は、撮影、保存、本部確認、SNS投稿素材の管理までを一度整えてから見直す。

### 店舗担当者が使う機能

店舗担当者は、原則として店舗側画面だけを使う。

現在の主な日常操作:

- 店舗コードとPINでログインする。
- 必要に応じて写真枠を選ぶ。
- わんちゃんを撮影する。
- 3枚の中から写真を選ぶ。
- 撮影担当者を選ぶ。
- 必要な場所だけ手動でモザイクする。
- 短い文字を写真上へ入れる。
- 完成画像を確認する。
- 印刷する。
- お客様からSNS掲載OKをもらった場合だけ保存する。

店舗担当者が直接やらない想定:

- DBを直接編集する。
- Supabase Storageの画像ファイルを直接消す。
- 本番用APIキーや環境変数を触る。
- 他店舗のデータを編集する。
- 店舗、担当者、枠のマスタを直接変更する。
- SNS投稿の最終管理を行う。

### 本部担当者が使う機能

本部担当者は、原則として `/admin` を使う。

現在の主な管理操作:

- 保存済み完成画像を一覧確認する。
- 保存済み完成画像の説明文を更新する。
- 保存済み完成画像を非表示、復帰する。
- 店舗マスタを編集する。
- 担当者マスタを追加、停止、編集する。
- 店舗フレームを登録、差し替えする。

後続フェーズで検討する管理操作:

- 残っているロゴ管理を削除または非表示にする。
- 投稿候補画像を選ぶ。
- SNS投稿素材を作成、保存する。
- 投稿文を確認、編集する。
- 投稿済み、保留、投稿不可などの状態を管理する。
- 完全削除やStorageファイル削除を行う。

## 削除・非表示の方針

店舗側、本部側ともに、最初は完全削除より非表示を優先する。

```text
非表示操作
→ assets.status を archived にする
→ 日常一覧には出さない
→ 本部側では確認・復帰できる
```

完全削除は、運用責任者または本部管理者だけが行う。
写真や記録を誤って消すと戻せないため、店舗側に完全削除を持たせない。

## 画面分離の方針

```text
/store または店舗側画面
店舗iPad用。撮影、枠選択、写真選択、担当者選択、編集、印刷、SNS掲載OK後の保存を行う。

/admin
本部管理用。保存画像、店舗、担当者、枠を管理する。将来はSNS投稿素材管理を追加する。
```

店舗側と本部側は同じSupabaseを使うが、画面と操作権限は分ける。
店舗側の機能追加が本部側の管理機能を壊さないようにし、本部側の投稿管理が店舗の撮影体験を重くしないようにする。

## GitHub

- ソースコードの正はGitHubの `main` とする。
- CodexはGitHub mainへ直接反映する方針とする。
- ローカルPC上の作業コピーやDropbox内ファイルを正本にしない。
- 店舗側へ引き渡す場合は、リポジトリの所有権移管または店舗側Organizationへの移動を検討する。
- 移管前に、管理者権限を持つユーザー、GitHub App、不要な外部アクセスを確認する。

## Vercel

- VercelはGitHubリポジトリから自動デプロイする。
- 本番プロジェクトは、可能であれば店舗側または運用主体のVercel Team配下へ移す。
- VercelのEnvironment Variablesに設定する値は `.env.example` と対応させる。
- 引き渡し時には、Production、Preview、Developmentの環境変数差分を確認する。
- `ADMIN_MAINTENANCE_PIN` はVercel側で管理し、GitHubへ値を記録しない。

## Supabase

- DB、Storage、店舗設定、担当者、枠、完成画像はSupabaseに集約する。
- ロゴは新規管理対象にせず、枠画像内のデザインとして扱う。
- Supabaseプロジェクトは、可能であれば店舗側または運用主体のOrganization配下へ移す。
- Service Role Keyはサーバー側だけで使い、ブラウザへ絶対に出さない。
- anon keyはブラウザで使う前提だが、RLSでアクセス範囲を必ず制限する。
- 引き渡し前に、RLSポリシー、Storageバケット、Auth設定、バックアップ方針を確認する。
- 本番DBへ適用したSQLは、必ず `supabase/migrations/` にも記録する。

## 環境変数

現時点で想定する主な値は以下。

```text
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY
ADMIN_MAINTENANCE_PIN
```

`NEXT_PUBLIC_` で始まる値はブラウザに公開される。秘密にしたい値には使わない。

## 引き渡し前チェックリスト

- GitHubリポジトリの所有者と管理者を確認する。
- GitHub Apps、外部連携、不要なアクセス権限を確認する。
- Vercelプロジェクトの所有Team、請求、環境変数を確認する。
- Supabaseプロジェクトの所有Organization、請求、APIキー、RLSを確認する。
- `.env.example` と実際のVercel環境変数が対応しているか確認する。
- SupabaseのStorageバケット、公開範囲、バックアップ方針を確認する。
- 本番データのバックアップと削除対応の運用者を決める。
- 店舗マスタ、担当者マスタ、枠画像の更新担当者を決める。
- 店舗側の日常操作担当者と、本部側の管理担当者を決める。
- 保存済み写真の非表示、完全削除、復旧のルールを決める。
- 緊急時にクラウド保存や本部管理画面の利用を止める手順を決める。
- Canon SELPHY / AirPrintの実機印刷手順を確認する。
- 次の運用者が、GitHub、Vercel、Supabaseだけを見て再開できるか確認する。
