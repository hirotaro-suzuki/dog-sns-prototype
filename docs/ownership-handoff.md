# 運用引き渡しメモ

このプロジェクトは、最終的に店舗または運用主体へ引き渡す前提で管理する。

## 基本方針

- GitHub、Vercel、Supabaseは、後から店舗側の管理アカウントへ移管しやすい構成にする。
- 個人アカウントにしか存在しない設定、環境変数、APIキー、画像ファイルを作らない。
- 本番環境の値はコードに書かず、VercelとSupabaseの管理画面で差し替えられるようにする。
- 引き渡し前に、所有者、請求、管理者、環境変数、バックアップ、緊急連絡先を一覧化する。
- 現場担当者が日常操作で迷わないことを優先し、店舗側画面と本部側画面の役割を分ける。

## 仮の運用分担

この分担は、撮影から保存、本部確認、SNS投稿までの流れ全体を一度作ったあとで見直す。
現時点では、後から引き渡しやすくするための仮方針として扱う。

### 店舗担当者が使う機能

店舗担当者は、原則として `/store` だけを使う。

想定する日常操作:

- 店舗コードとPINでログインする。
- 撮影ごとに担当者を選ぶ。
- わんちゃんを撮影する。
- 3枚の中から写真を選ぶ。
- わんちゃん情報を入力する。
- 必要な場所だけ手動でモザイクする。
- 完成画像を確認する。
- 印刷する。
- お客様からSNS掲載OKをもらった場合だけ保存する。
- 保存済み写真を日付別に確認する。
- 日付で検索する。
- 保存済み写真を再印刷する。
- 誤って保存した写真を非表示にする。

店舗担当者が直接やらない想定:

- DBを直接編集する。
- Storageの画像ファイルを直接消す。
- 本番用APIキーや環境変数を触る。
- 他店舗のデータを編集する。
- SNS投稿の最終管理を行う。

### 本部担当者が使う機能

本部担当者は、原則として `/admin` を使う。

想定する管理操作:

- 店舗を追加、停止、編集する。
- 担当者を追加、停止、編集する。
- 店舗ロゴを登録、差し替えする。
- 店舗フレームを登録、差し替えする。
- 保存済み完成画像の一覧を見る。
- 投稿候補画像を選ぶ。
- 投稿文を確認、編集する。
- 投稿済み、非表示、保留などの状態を管理する。
- 誤登録データを整理する。

本部担当者だけに寄せたい操作:

- 完全削除。
- Storageファイル削除。
- 本番店舗設定の大きな変更。
- SNS連携設定。
- 管理者権限の追加、削除。

### 削除の仮方針

店舗側の「削除」は、最初は完全削除ではなく非表示扱いにする。

例:

```text
店舗側で削除ボタンを押す
→ assets.status を archived にする
→ 店舗側の日常一覧には出さない
→ 本部側では確認できる
```

完全削除は、運用責任者または本部管理者だけが行う。
写真や記録を誤って消すと戻せないため、最初から店舗側に完全削除を持たせない。

## 画面分離の仮方針

```text
/store
店舗iPad用。撮影、編集、印刷、SNS掲載OK後の保存、保存済み写真の確認、再印刷を行う。

/admin
本部管理用。店舗、担当者、ロゴ、フレーム、保存画像、投稿候補、投稿文、状態管理を行う。
```

店舗側と本部側は同じSupabaseを使うが、画面と操作権限は分ける。
店舗側の機能追加が本部側の管理機能を壊さないようにし、本部側の投稿管理が店舗の撮影体験を壊さないようにする。

## まだ確定しないこと

以下は、撮影、保存、本部確認、SNS投稿までの流れを一度つないでから改めて決める。

- 店舗側で保存済み写真を何日分まで表示するか。
- 店舗側で他店舗データを一切見せないための権限設計。
- 本部管理者のログイン方法。
- 店舗担当者を現場画面から追加できるようにするか。
- 削除、非表示、投稿不可、保留などの状態名。
- SNS投稿をどこまで自動化するか。
- 投稿文生成にAIを使う場合の確認手順。
- 完成画像と投稿文の最終承認者。

## GitHub

- ソースコードの正はGitHubの `main` とする。
- 店舗側へ引き渡す場合は、リポジトリの所有権移管または店舗側Organizationへの移動を検討する。
- 移管前に、管理者権限を持つユーザーと不要な外部アクセスを確認する。

## Vercel

- VercelはGitHubリポジトリから自動デプロイする。
- 本番プロジェクトは、可能であれば店舗側または運用主体のVercel Team配下へ移す。
- VercelのEnvironment Variablesに設定する値は `.env.example` と対応させる。
- 引き渡し時には、Production、Preview、Developmentの環境変数差分を確認する。

## Supabase

- DB、Storage、Auth、店舗設定、完成画像はSupabaseに集約する。
- Supabaseプロジェクトは、可能であれば店舗側または運用主体のOrganization配下へ移す。
- Service Role Keyはサーバー側だけで使い、ブラウザへ絶対に出さない。
- anon keyはブラウザで使う前提だが、RLSでアクセス範囲を必ず制限する。
- 引き渡し前に、RLSポリシー、Storageバケット、Auth設定、バックアップ方針を確認する。

## 環境変数

現時点で想定する値は以下。

```text
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY
```

`NEXT_PUBLIC_` で始まる値はブラウザに公開される。秘密にしたい値には使わない。

## 引き渡し前チェックリスト

- GitHubリポジトリの所有者と管理者を確認する
- Vercelプロジェクトの所有Team、請求、環境変数を確認する
- Supabaseプロジェクトの所有Organization、請求、APIキー、RLSを確認する
- `.env.example` と実際のVercel環境変数が対応しているか確認する
- 本番データのバックアップと削除対応の運用者を決める
- 店舗マスタ、担当者マスタ、ロゴ、枠画像の更新担当者を決める
- 店舗側の日常操作担当者と、本部側の管理担当者を決める
- 保存済み写真の非表示、完全削除、復旧のルールを決める
- 緊急時にSNS保存やクラウド保存を止める手順を決める
