## 3. 2026-07-11にユーザーと確認した判断

### 3-1. 店舗側の保存API認証

監査では、店舗ログイン後に署名トークンや正式セッションがなく、`/api/assets` が `storeId` と `staffId` を受け取って保存している点を指摘しました。

これについて、ユーザーは次の運用判断をしました。

- 現状のままでよい。
- 店舗担当者情報は、厳格な権限制御ではなく、単なるメモに近い扱いである。
- 現場スタッフの操作を軽くすることを優先する。
- 将来、運用方針や扱う情報の重要度が変わらない限り、この点だけを理由に店舗フローを複雑化しない。

したがって、この項目は技術上のリスクとして記録は残しますが、現時点では「合意済み・対応不要」と扱います。

ただし、次の問題はこの合意とは別に検討します。

- 認証なし保存を大量に繰り返されるStorage消費
- 画像形式や画像寸法の検証不足
- 二重保存、連続保存、機械的な大量送信への対策
- 公開Storage上の写真URL推測

### 3-2. Admin PIN

- Admin PINは、将来もう少し強くする意向がある。
- 現時点では大きく作り直さない。
- 本格的な管理者認証は、引き渡し後に先方が対応する候補とする。

引き渡し時には、少なくとも次を先方へ明示します。

- 現在は静的な共通PINであること
- PIN試行回数制限や利用者別権限がないこと
- PINを変更するにはVercel環境変数の更新が必要なこと
- Supabase Auth等へ移行する場合の対象API一覧

## 6. 2026-07-10修正の評価

### `store_frames.is_active`

- 保存APIから参照削除済み
- TypeScript型から削除済み
- 現行schemaから削除済み
- `20260708_store_frame_slots.sql` で既存DBから削除する履歴あり

この項目はコード・DB定義上は解消済みです。

### 停止店舗削除

確認できた安全策:

- 稼働中店舗は削除しない
- 保存写真が1件でもあれば削除しない
- `assets.store_id` は `ON DELETE RESTRICT`
- `staff_members` と `store_frames` は `ON DELETE CASCADE`
- 最終DELETEでも `is_active = false` を条件にする
- 写真が並行追加された場合は外部キーで削除を止める
- Storage保存先の店舗コード衝突を確認する
- Storageを削除後に再走査する

残る問題:

- DB店舗削除後にStorage削除が失敗しても、警告付き成功になる
- 残ったStorageファイルを後から再試行する記録がない
- 店舗削除と同時期に完成画像保存が走り、DB登録失敗後のStorage掃除にも失敗すると `final-images` に孤児が残り得る

DB上の写真保護と競合対策は良好ですが、DBとStorageを合わせた完全な原子性はありません。

### Security Advisor

コードとSQLでは、次を確認しました。

- `public.set_updated_at()` の `search_path=pg_catalog`
- `public.rls_auto_enable()` が存在する場合の同設定
- `public`、`anon`、`authenticated` からの直接実行権限剥奪
- migrationがトランザクションで囲まれている

本番SupabaseでWarnings 0件になったことは文書に記録されていますが、今回の監査では実Supabaseへ接続して再確認していません。

## 7. 新規に見つかった主な項目

### 完成画像Storageがpublicで、パスを推測しやすい

`final-images` bucketはpublicで、保存パスは次の規則です。

```text
[store_code]/[year]/[month]/[day]/[store_code]-[yyyymmdd]-[3桁連番].jpg
```

Supabase URLと店舗コードを知っている場合、未投稿写真を順番に推測できる可能性があります。保存対象はSNS掲載同意済み写真ですが、公開タイミングと公開範囲を改めて検討する候補です。

### 保存APIの画像・回数制限

- Base64文字数の上限はある
- JPEGの実体確認、画像寸法確認、回数制限はない
- 同じ操作の二重送信をサーバー側で防がない
- 認証方式を現状維持する場合でも、Storage消費対策は別途検討できる

### Admin PIN

静的PINを各APIが直接比較します。正式セッション、回数制限、利用者別権限、操作監査はありません。

この点は2026-07-11の判断により、引き渡し後の対応候補として記録します。

### DBとTypeScript型の差

TypeScript型には `dog_name`、`dog_breed`、`dog_age`、`staff_comment` が残っていますが、現行 `schema.sql` の `assets` にはありません。

### 管理画面の並行読込状態

写真、店舗、担当者の複数通信が、共通の `isLoading` と `message` を操作します。先に終わった通信が読込中表示を解除したり、別の通信結果がメッセージを上書きする可能性があります。

### 依存バージョンの固定

`package.json` は `^` 付きバージョンで、mainにlockfileがありません。今回の環境ではNext.js 15.5.20を使用しましたが、将来のVercelインストールで別バージョンになる可能性があります。

## 8. `.next`、`.claude/`、`package-lock.json`

### `.next` がlint対象になる原因

- `npm run lint` が `eslint .`
- ESLint flat configに `.next/**` の除外がない
- `.gitignore` の `/.next/` はflat configへ自動反映されない
- `.next/types/routes.d.ts` 1ファイルだけで11件のlintエラーを再現した

対応候補:

1. `lint` を `eslint src` にする
2. ESLint configにも `.next/**`、`out/**` 等の除外を明記する

### `.claude/`

- 秘密情報は見つからなかった
- 絶対パス、sudo、pip、git操作等の環境固有許可が含まれる
- 現在のままcommitしない
- 当面は `/.claude/` を `.gitignore` に加える案が妥当
- 共通ルールは `AGENTS.md` と `CLAUDE.md` に置く

### `package-lock.json`

- 未追跡でGit履歴にない
- 2026-07-07作成、lockfileVersion 2
- 今回の実行環境と同じNext.js 15.5.20等を記録
- 秘密情報は見つからなかった
- 削除するより、別作業でクリーン検証してから正式に追跡する案を推奨
- 出所がGitで追えないため、そのままcommitしない
