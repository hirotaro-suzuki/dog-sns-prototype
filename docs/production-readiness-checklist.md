# 本番前チェックリスト

最終更新: 2026-07-06

この文書は、デモまたは本番運用へ入る前に、テストデータや仮設定を整理するためのチェックリストです。

毎回読む文書ではありません。デモ前、本番前、またはSupabase上のデータを整理するときに使います。

前提:

- ソースコードと文書は GitHub `main`
- 実行、ビルド、公開確認は Vercel
- DB、Storage、Auth、店舗設定、担当者、枠、完成画像は Supabase
- ローカルPC、Dropbox、手元コピーは正本にしない

Supabaseの詳しい構造は `supabase/schema.sql` と `docs/supabase-handoff.md` を参照します。

## 1. 本番前に決めること

- 本番で使う店舗一覧。
- 店舗ごとの正式な店舗コード、ログインコード、PIN設定方法。
- 本番で使う担当者一覧、表示名、役割、並び順。
- 店舗ごとの正式な写真枠と標準枠。
- テスト期間中に保存した写真を完全削除するか、非表示で残すか。
- Storage内のテスト画像を削除するか、残すか。
- 本部PIN `ADMIN_MAINTENANCE_PIN` を誰が管理するか。
- 本番前にバックアップを取るか。取る場合は誰が保管するか。
- Canon SELPHY / AirPrintの実機印刷手順を誰が確認するか。

## 2. 整理対象

### DB側

- `stores`: 店舗マスタ
- `staff_members`: 店舗ごとの担当者マスタ
- `store_frames`: 店舗ごとの写真枠マスタ
- `assets`: 保存済み完成画像の管理情報

### Storage側

- `final-images`: SNS掲載OK後に保存した完成画像
- `store-assets`: 店舗フレームなどの画像素材

DB上の `assets` を削除しても、Storage上の画像ファイルが自動で消えるとは限りません。DB整理とStorage整理は別作業として扱います。

## 3. 消してよい候補

本番前に、必要に応じて以下を整理します。

- テスト店舗
- テスト担当者
- 仮枠、テスト枠、社長確認に出せない枠
- テスト期間中に保存した写真のDBレコード
- `final-images` 内のテスト完成画像
- `store-assets` 内の不要な仮枠画像

削除が不安な場合は、まず `is_active = false` や `assets.status = archived` による非表示を優先します。

## 4. 消さないもの

本番前整理では、基本的に以下は消しません。

- DBテーブル構造
- migration履歴としてGitHubに残したSQL
- Storage bucketそのもの
- Vercel環境変数
- Supabase APIキーやAuth設定
- GitHub上のコードと文書

## 5. 実行前チェック

Supabase SQL Editorなどで削除や整理を行う前に確認します。

- 削除対象がテストデータだけである。
- 本番用の店舗、担当者、枠の再投入データが用意されている。
- 削除後に店舗ログインできるデータが残る、または再投入できる。
- `assets` を削除する場合、対応するStorage画像をどうするか決めている。
- `store_frames` を削除する場合、対応する枠画像をStorageに残すか消すか決めている。
- 実行するSQLまたは手順をGitHub上に残している。
- 実行後の確認項目が決まっている。

## 6. 実行後チェック

整理後は、以下を確認します。

- 本番店舗だけが有効になっている。
- 店舗ログインコードとPINでログインできる。
- 店舗ごとに担当者が正しく表示される。
- 無効な担当者が店舗側の担当者選択に出ない。
- 店舗ごとに有効な枠だけが表示される。
- 標準枠が1つに決まっている。
- テスト写真が本部写真タブに出ない。
- 必要なら `final-images` のテスト画像も削除済みである。
- `/admin` が本部PINで開ける。
- Vercelの最新デプロイが `success` である。
- iPad Safariで店舗ログイン、撮影、担当者選択、枠選択、印刷、同意後保存まで確認できる。

## 7. 本番前整理と同時に先走らないこと

以下は、本番前整理と同時に実装しません。

- SNS投稿
- 自動投稿
- 投稿本文生成
- Instagram連携
- Meta API連携
- 複雑な管理者権限
- 店舗側からの完全削除

## 8. 次に必要になりそうな作業

- 現在のSupabase実データ件数を確認するSQLを作る。
- 本番前クリア用SQL案を作る。
- Storage削除手順を整理する。
- 本番初期データ投入用SQLまたは手順を作る。
- 正式店舗、正式担当者、正式枠の一覧を店舗側と確認する。
