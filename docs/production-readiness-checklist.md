# 本番前作業項目

最終更新: 2026-07-05

この文書は、テスト期間のデータや仮設定を整理し、本番運用へ入る前に確認する項目をまとめる。

このプロジェクトでは、GitHub、Vercel、Supabaseを正とする。ローカルPC、Dropbox、手元フォルダを正本にしない。

## 目的

テスト期間中は、店舗、担当者、枠、保存済み写真に仮データが入る。

本番開始前には、テストデータと本番データが混ざらないように、Supabase上のDBとStorageを整理し、正式な店舗、担当者、枠だけで開始できる状態にする。

## 現在Supabaseにある主な管理対象

GitHub main上の `supabase/schema.sql` では、主に以下を扱う。

### DBテーブル

- `stores`: 店舗マスタ
- `staff_members`: 店舗ごとの担当者マスタ
- `store_frames`: 店舗ごとの写真枠マスタ
- `assets`: 保存済み完成画像の管理情報
- `admin_users`: 将来の管理者用テーブル

### Storage bucket

- `final-images`: SNS掲載OK後に保存した完成画像
- `store-assets`: 店舗フレームなどの画像素材

## タブに出ていない重要項目

### 店舗ログイン情報

`stores` には、店舗タブに出している項目以外に以下がある。

- `login_code`: 店舗ログイン用コード
- `pin_hash`: 店舗PINのハッシュ

店舗PINそのものは平文で持たず、ハッシュとして保存する。

### 本部ログイン情報

本部メンテナンス画面 `/admin` は、現時点では主にVercel環境変数 `ADMIN_MAINTENANCE_PIN` で開く。

`ADMIN_MAINTENANCE_PIN` の値はGitHubへ記録しない。

### 互換用または後続整理予定の項目

`stores` には、現在の画面から外した項目や古い互換用項目が残っている。

- `logo_url`
- `frame_url`
- `theme_color`
- `sns_display_name`
- `instagram_account`
- `default_hashtags`
- `address`
- `phone`
- `business_hours_note`

ロゴは単独素材ではなく、写真枠画像内に含める方針へ変更済み。ロゴ関連項目は、枠画像への移行が済んでから削除または非表示を検討する。

### 保存済み写真のStorage情報

`assets` には、完成画像そのものではなく、Storage上の保存先情報を持つ。

- `final_processed_url`
- `final_storage_bucket`
- `final_storage_path`

DB上の `assets` を消しても、Storage上の画像ファイルが自動で消えるとは限らない。DB整理とStorage整理は別作業として扱う。

## 本番前に決めること

本番開始前に、少なくとも以下を決める。

- 本番で使う店舗一覧
- 店舗ごとの正式な店舗コードとログインコード
- 店舗ごとのPIN設定方法
- 本番で使う担当者一覧
- 担当者の表示名、役割、並び順
- 店舗ごとの正式な写真枠
- 店舗ごとの標準枠
- テスト期間中に保存した写真を完全削除するか、非表示で残すか
- Storage内のテスト画像を削除するか
- 本部PIN `ADMIN_MAINTENANCE_PIN` を誰が管理するか
- 本番前にバックアップを取るか、取る場合は誰が保管するか

## 本番前データ整理方針

### 店舗と担当者

通常運用では、店舗や担当者は物理削除せず、`is_active = false` で無効化する。

ただし、本番開始前のテストデータについては、正式運用データと混ざらないように、全削除して本番データを入れ直す選択肢もある。

本番前に全削除する場合も、実行前に対象と復旧可否を確認する。

### 枠

仮枠、テスト枠、社長確認に出せない枠は、本番開始前に整理する。

本番開始時は、各店舗ごとに以下が分かる状態にする。

- 有効な枠
- 標準枠
- 枠画像URL
- 日付表示の有無、位置、文字サイズ、色

### 保存済み写真

テスト期間に保存した写真は、本番データと混ざりやすい。

本番開始前に、以下のどちらにするか決める。

- `assets` と `final-images` のテスト画像を削除して、きれいな状態で開始する
- テスト写真は `archived` として非表示にし、後で確認できるように残す

写真は個人情報に近い扱いになる可能性があるため、Storage上のファイル削除も忘れずに検討する。

## 本番前クリアの考え方

本番前に全データをクリアして始めることは可能。

ただし、DB構造、制約、Storage bucket、環境変数は消さず、基本的には中身だけを整理する。

クリア対象候補:

- `assets`
- `staff_members`
- `store_frames`
- `stores`
- `final-images` 内のテスト完成画像
- `store-assets` 内の不要な仮枠画像

消さないもの:

- DBテーブル構造
- migration履歴としてGitHubに残したSQL
- Storage bucketそのもの
- Vercel環境変数
- Supabase APIキーやAuth設定
- GitHub上のコードと文書

## 実行前チェック

実際にSupabase SQL Editorなどで削除作業を行う前に、以下を確認する。

- 削除対象がテストデータだけである
- 本番用の店舗、担当者、枠の再投入データが用意されている
- 削除後に店舗ログインできるデータが残る、または再投入できる
- `assets` を削除する場合、対応するStorage画像も扱う
- `store_frames` を削除する場合、対応する枠画像をStorageに残すか消すか決めている
- 実行するSQLまたは手順をGitHub上の文書に残している
- 実行後の確認項目が決まっている

## 実行後チェック

本番前整理後は、以下を確認する。

- 本番店舗だけが有効になっている
- 店舗ログインコードとPINでログインできる
- 店舗ごとに担当者が正しく表示される
- 無効な担当者が店舗側の担当者選択に出ない
- 店舗ごとに有効な枠だけが表示される
- 標準枠が1つに決まっている
- テスト写真が本部写真タブに出ない
- 必要なら `final-images` のテスト画像も削除済みである
- `/admin` が本部PINで開ける
- Vercelの最新デプロイが `success` である
- iPad Safariで店舗ログイン、撮影、担当者選択、枠選択、印刷、同意後保存まで確認できる

## まだ実装しないこと

本番前整理と同時に、以下を先走って実装しない。

- SNS投稿
- 自動投稿
- 投稿本文生成
- Instagram連携
- Meta API連携
- 複雑な管理者権限
- 店舗側からの完全削除
