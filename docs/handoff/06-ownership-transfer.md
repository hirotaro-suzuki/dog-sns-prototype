# 権利・アカウント引き継ぎ手順書（オーナー向け）

最終更新: 2026-07-20

この資料は、システムの「持ち主」を引き受けるオーナーのためのものです。
（技術担当向けの正確な記録は `docs/ownership-handoff.md` にあり、そちらが技術的な正本です。この資料はその平易版です。）
何の持ち主になるのか、何にお金がかかるのか、どんな順番で引き継ぐのかを説明します。
技術的な作業の中身は、技術担当が資料7を見ながら行うので、オーナーが覚える必要はありません。

## 1. 引き渡す4つのアカウント

このシステムは、4つのサービスの上に載っています（絵は資料2）。それぞれに「持ち主のアカウント」があり、現在はすべて開発者（鈴木）の個人アカウントです。引き継ぎでは、これをオーナー側のアカウントへ移します。

| サービス | 持ち主になると何を握るか | 移し方 |
|----------|--------------------------|--------|
| GitHub（設計図の保管庫） | プログラムと説明書の正本。これが失われると修理・改良ができなくなる | リポジトリ（保管庫の箱）の所有権移管 |
| Vercel（店先） | 公開画面そのもの。止めると全店舗で使えなくなる | プロジェクトの移管、または新アカウントで作り直し |
| Supabase（金庫） | 保存写真と全データ。**一番大事** | プロジェクトの所有組織を変更 |
| メールアドレス | 上の3つの登録先。パスワード再設定の受け皿 | オーナー側で管理するアドレスを1つ決める |

**おすすめ**: 個人の私用アドレスではなく、お店の事業用メールアドレスを1つ決めて、3つのサービスすべてをそこに登録することです。担当者が変わっても引き継げます。

## 2. お金の話（有料契約）

2026年7月時点の目安です。**為替やプラン改定で変わるため、契約前に必ず最新の料金ページを確認してください。**

| サービス | 無料のままでよいか | 有料が必要な理由 | 目安 |
|----------|--------------------|-------------------|------|
| GitHub | 無料でよい | ―― | 0円 |
| Vercel | **有料が必要** | 無料プランは「商用利用不可」の決まり。お店の営業で使うなら有料プラン（Pro）が条件 | 月20米ドル程度 |
| Supabase | **有料を強くおすすめ** | 無料プランは、しばらく使わないと金庫が一時停止する・バックアップが弱い。お客様の写真を預かる以上、毎日の自動バックアップがある有料プラン（Pro）が安心 | 月25米ドル程度 |
| メール | 無料でよい | ―― | 0円 |

合計で、**月に45米ドル前後（7,000円前後）** が続くお金の見当です。

**大事な順序**: 有料契約は、所有権を移す「前」に済ませてください。契約しないまま持ち主だけ替わると、無料プランの制限（商用利用不可・金庫の一時停止）を持ち主として抱えることになります。

## 3. 引き継ぎの順番

全体の流れです。左の番号順に進めます。オーナー自身の作業は ★ の2つだけで、あとは技術担当と開発者が行います。

1. ★ **事業用メールアドレスを決める**（新設でも既存でもよい）
2. ★ **VercelとSupabaseの有料契約を行う**（技術担当が画面操作を手伝う）
3. 技術担当をGitHub・Vercel・Supabaseへ「メンバー」として招待し、開発者と両方が入れる共有期間を作る
4. **本番切替**: テスト用のデータを削除し、本物の店舗・担当者・写真枠を登録する（技術担当と開発者が一緒に行う。手順書あり）
5. 所有権をオーナー側アカウントへ移す
6. **秘密の値をすべて作り直す**（下の4章）
7. 申し送り事項（下の5章）を確認して、引き継ぎ完了。開発者は手を離れる

## 4. 引き継ぎ時に必ず作り直すもの（合鍵の交換）

家を引き渡すとき鍵を交換するのと同じで、次のものは引き継ぎ完了時に技術担当が新しく作り直します。開発者の手元に古い鍵が残らないようにするためです。

- 本部PIN（/admin に入る暗証番号）
- 各店舗のログインPIN
- Supabaseの合鍵類（APIキーと呼ばれる、プログラム同士をつなぐ鍵）
- 各サービスのパスワード（オーナー側アカウントにした時点で自動的に別物になります）

## 5. 持ち主として知っておくべきこと（申し送り）

- **お客様の写真を預かる責任が移ります。** お客様から「消してほしい」と言われたら、本部画面から完全削除で応じます（操作は資料4）。この対応義務は持ち主のものです。
- **保存写真は「住所を知っていれば見られる」保管方法です。** 写真の保管場所には鍵がかかっておらず、写真のURL（長くて推測されにくい住所）を知っている人は誰でも見られます。SNS掲載OKをもらった写真しか保存しない決まりなので今すぐの危険は小さいですが、将来、技術担当に鍵付きへの変更を検討してもらう項目として申し送ります。
- **本部PINは1つだけで、入力回数の制限がありません。** 今のままでも運用できますが、これも将来の強化候補として技術担当へ申し送ります。
- SNSへの自動投稿機能はありません。「自動でSNSに載る」とお客様や店舗に説明しないでください。

技術的な詳しい内容は、技術担当向けの資料7と、docs/ フォルダの既存文書に記録されています。
