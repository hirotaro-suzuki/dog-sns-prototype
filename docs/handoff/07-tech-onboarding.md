# 技術引き継ぎの入口（技術担当向け）

最終更新: 2026-07-20

この資料は、システムの技術面を引き継ぐ人のためのものです。
AIを使った開発（いわゆるバイブコーディング）の経験がある前提で書いていますが、プロのエンジニアである必要はありません。
このシステムは、開発の最初からAIと一緒に作業する前提で文書やルールが整備されています。あなたも同じやり方で引き継げます。

## 1. 最初の一歩

1. GitHubアカウントを作る（無料でよい）。
2. 開発者から `dog-sns-prototype` リポジトリへの招待を受け取り、中を見られることを確認する。
3. リポジトリの「Code」ボタン →「Codespaces」で作業環境をブラウザ上に起動する。**開発作業はこのCodespacesの中だけで行う。** 自分のPCへコピーして作業しない（正本が分からなくなるのを防ぐため）。
4. Codespacesのターミナルで `npm install` → `npm run build` が通ることを確認する。これが「壊していないこと」の基本チェック。

## 2. このリポジトリでのAIとの作業ルール

リポジトリ直下の `CLAUDE.md`（Claude Code用）と `AGENTS.md`（Codex用）に作業規則がまとまっており、AIは起動時に自動で読みます。人間側が守らせることは、次の4つです。

1. **実装の前に、AIに計画を出させて自分が納得してから進める。** 「いきなり書き始めない」がこのプロジェクトの一番のルール。
2. **文書の地図に従わせる。** まず `docs/START_HERE.md` と `docs/DOC_INDEX.md`。AIにリポジトリ全体を手当たり次第探索させない。
3. **データベースを変える作業は、実行したSQLを必ず `supabase/migrations/` に記録させる。** 「本番の金庫に何をしたか」が後から追えることが生命線。
4. **こまめにコミット・pushさせる。** Codespacesは予告なく止まることがある。

## 3. AIに任せてよい作業と、必ず止まって確認する作業

| AIに任せてよい | 必ず自分で止まって確認してから |
|----------------|-------------------------------|
| 画面の文言・見た目の修正 | データベースの構造変更（テーブル・列の追加や削除） |
| バグの調査と修正案の作成 | データの削除全般（写真・店舗・Storage内のファイル） |
| ビルド確認、コミット、push | 秘密の値の操作（PIN、APIキー、環境変数） |
| 文書の更新 | Supabase・Vercelの設定変更 |

右の列は「AIが提案 → 自分が中身を理解 → 自分の判断で実行」の順を守ってください。
**本番のデータベースで実験しないこと。** 迷ったら、まず読み取り（SELECT）だけで状況を確認する癖をつけてください。

## 4. 読む文書の順番

1. `docs/START_HERE.md` — 作業ルールの正本
2. `docs/DOC_INDEX.md` — 全文書の地図。ここから必要なものだけ読む
3. `docs/ownership-handoff.md` — 引き渡しの技術方針、環境変数の一覧
4. `docs/supabase-handoff.md` — データベースとStorageの引き継ぎメモ（migrationの適用状況はここが正）
5. `dog_sns_design.md` — 画面仕様とDB設計の正本
6. `docs/audit-2026-07-13.md` — 全コードを1件ずつ点検した監査記録。システムの弱点が正直に書いてある

## 5. 仕組みの要点（1分版）

- 中身はNext.js（React + TypeScript）のアプリ。コードは `src/`、画面は店舗側フローと `/admin` の2系統。
- `main` ブランチにpushすると、Vercelが自動で本番を建て直す。**つまりpush＝本番反映。** ビルド確認をしてからpushする。
- データはすべてSupabase。DBのテーブル定義は `supabase/schema.sql`、変更履歴は `supabase/migrations/`。Storageは `final-images`（完成写真）と `store-assets`（枠画像）の2つ。
- 秘密の値（APIキー、本部PIN）はVercelの環境変数にあり、GitHubには書かない。一覧は `docs/ownership-handoff.md`。

## 6. 申し送り事項（既知の弱点。今すぐ直さなくてよい）

前任の開発者とAIが監査で洗い出し、「実害が出るまで着手しない」「引き渡し後の課題」と判断済みの項目です。詳細は `docs/audit-2026-07-13.md` の番号を参照。

1. **枠の並べ替え・標準枠変更がトランザクションになっていない（#16）** — 複数のDB更新を一括で行っていないため、途中で失敗すると順序がずれる可能性がある。枠画像のStorageに使われないファイルが残ることもある。実害が出たら着手。
2. **完成写真の保管場所（final-images）が公開設定（#18）** — URLを知っていれば誰でも見られる。SNS掲載OKの写真しか保存しない設計なので緊急性は低いが、将来は鍵付き（署名付きURL）への変更を検討。
3. **本部PINが固定1つ・回数制限なし** — 強化（回数制限、担当者別の権限など）は引き渡し後の課題として合意済み。
4. **店舗の保存APIは正式な認証を持たない** — 2026-07-11に「現場優先で現状維持」と合意済み。技術リスクとして記録のみ。

## 7. 本番切替であなたが担当する技術作業

引き継ぎの途中で「テストデータを消して本物のデータを入れる」作業を、前任の開発者と一緒に行います。手順書は `docs/production-readiness-checklist.md`。

- 本物の店舗・担当者・枠の登録は **本部画面 /admin でできる**（あなたの実地訓練を兼ねる）
- /admin でできないのは: Storage内のテスト画像の整理、デモ店舗（DEMO_STORE）の残骸確認、`public/store-frames/` 内の使われていない枠SVGの整理。ここだけSupabaseとリポジトリを直接触る

## 8. 困ったとき

- まず `docs/handoff/05-troubleshooting.md`（店舗・本部向けの手引き）で症状を確認。
- 技術的に行き詰まったら、AIに `docs/DOC_INDEX.md` から関連文書を読ませて相談する。過去の作業ログ（docs/session-\*.md）に、ほとんどの画面の変更経緯が残っている。
- それでも解決しない場合に前任の開発者へ連絡してよいか・いつまでよいかは、引き渡し時に口頭で取り決める。
