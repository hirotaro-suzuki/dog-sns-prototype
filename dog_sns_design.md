# 店舗分散型SNS自動投稿システム 設計・開発方針

このファイルは、新しいスレッドや別作業者へ開発を引き継ぐための最重要ドキュメントです。開発前に必ずこのファイルを読み、ここに書かれた方針を最優先すること。

## 0. 最重要方針: ローカルに何も持たない

このプロジェクトでは、ユーザーのWindowsローカル環境にソースコード、依存関係、ビルド環境、実行環境を持たない方針を最優先とする。

- ソースコードはGitHubに集約する
- 実行・ビルド・プレビューはVercelで行う
- DB、Storage、Auth、店舗設定、完成画像はSupabaseに集約する
- ローカルPCでは `npm install`、`npm run dev`、Next.js初期化、依存関係インストールを原則行わない
- Codexは必要な変更をGitHubへ直接Pushする
- ユーザーはVercel上のURLをiPad Safariで確認する
- ローカルに一時的な開発ファイルや複製リポジトリを作ることも避ける

ローカル環境で詰まった場合に回避するのではなく、最初からクラウド直結の開発フローを正とする。

## 1. プロジェクトの目的

東京・軽井沢に複数店舗を展開する、わんちゃん同伴OKのイタリアンレストラン向けに、「今日のわんちゃん」写真を店頭で撮影・加工・保存し、将来的にSNSへ自動投稿できる仕組みを作る。

最終目標は、現場スタッフの負担を増やさず、愛犬家コミュニティへのブランド露出を継続的に高めること。

成功条件は、飲食店の忙しい現場でも迷わず使えること。ITに詳しくないスタッフでも、iPadだけで撮影、写真選定、画像加工、プリント、最終合意、保存まで完了できる体験を優先する。

## 2. 全体アーキテクチャ

システムは3つのレイヤーに分ける。

1. インプット: 現場のiPad Webアプリ
   店頭で複数枚撮影し、お客様と1枚を選び、日付・店名・必要なモザイクを合成して、記念プリントとクラウド保存を行う。フェーズ1の主対象。

2. プロセス: 本部の管理画面
   各店舗から上がってきた加工済み素材を確認し、投稿対象を選ぶ。将来的にGemini APIなどで投稿文を生成する。

3. アウトプット: ManusなどによるSNS投稿
   完成した画像と文章を、指示書通りにInstagram等へ投稿する。後続フェーズの対象。

各レイヤーは疎結合にする。フロント画面、画像加工、DB通信、AI文章生成、Manus連携、SNS投稿を密結合させない。

## 3. 確定技術スタック

- フロントエンド: Next.js
- ホスティング: Vercel
- DB: Supabase Database
- Storage: Supabase Storage
- Auth: Supabase Auth
- 画像加工: ブラウザ内の Canvas API
- 顔検出・モザイク下ごしらえ: 自動顔判定は原則使わず、スタッフによる手動モザイクを優先する
- 印刷: Web標準の `window.print()` と CSS `@media print`
- 対象端末: iPad Safari
- 対象プリンター: Canon SELPHY、AirPrint、写真L版、フチなし想定

すべての接続情報は初日から環境変数で読む。DB URL、Storage URL、APIキー、Supabase anon key、将来のAI APIキーなどをコードに直書きしてはいけない。

## 4. 現在のフェーズ0でできていること

Vercel上のプロトタイプで、以下の流れが動作確認済み。

1. iPad Safariでカメラを起動する
2. 最大3枚まで撮影する
3. 撮影画像をクラウドへ送らず、ブラウザメモリ内のBlob/Object URLだけで一時保持する
4. 3枚から1枚を選ぶ
5. わんちゃんの名前、犬種、犬齢を入力する
6. Canvasで日付、仮店舗名、仮ロゴ、犬情報を合成する
7. 写真だけをタッチで移動、拡大縮小、回転する
8. 必要な場所だけ手動モザイクする
9. 直前のモザイクだけ取り消す
10. 完成画像を生成してプレビューする

フェーズ0ではSupabaseへ保存しない。ブラウザメモリ内だけで撮影データの保持と破棄を確認する。

## 5. 次に実装する方針: 店舗ログインと担当者選択

次の開発テーマは、店舗ログイン、担当者選択、店舗ごとの写真枠・ロゴ・表示設定の連携である。

### 5.1 店舗ログイン

iPadは個人ではなく「店舗」としてログインする。

- 店舗コード + PIN、またはSupabase Authの店舗用アカウントでログインする
- 店舗ログインは最初に一度だけ行う想定
- ログイン後、店舗設定をSupabaseから取得する
- 店舗設定には、店舗名、店舗コード、ロゴ、写真枠、表示色、印刷テンプレート、担当者一覧を含める
- iPadは店舗に紐づく端末として扱う

個人スタッフごとのメールアドレス・パスワードログインは、現場負担が重いため最初は採用しない。

### 5.2 担当者選択

担当者はログインではなく、撮影ごとに選ぶメタ情報として扱う。

- 当日、同じiPadを複数名で使う可能性がある
- 撮影画面上で担当者を選択できるようにする
- 担当者は大きなボタンまたはセグメントコントロールで選ぶ
- 撮影のたびに担当者を切り替えられる
- 保存時には、その時点で選ばれていた `staff_id` と `staff_display_name` を素材データへ持たせる

想定UI:

```text
本日の担当
[山田] [佐藤] [鈴木] [店長]
```

フェーズ1では、担当者の追加・編集はSupabase側で行い、現場画面では選ぶだけにする。現場で自由追加できる機能は、必要性が見えてから検討する。

### 5.3 店舗ごとの写真枠・ロゴ

店舗ごとの表示情報はコードに書かず、Supabaseの店舗マスタから取得する。

- 撮影画面の半透明フレーム
- Canvas合成時の店舗名
- Canvas合成時の店舗ロゴ
- 印刷用テンプレート
- 表示カラー

これらは、ログイン中の店舗設定から動的に反映する。ロゴやフレームが未設定の場合は、画面が壊れないように仮表示へフォールバックする。

## 6. Supabaseデータ設計案

最初に必要なテーブルは以下。

### stores

店舗マスタ。

- `id`: UUID
- `store_code`: 店舗コード
- `store_name`: 正式店舗名
- `display_name`: 画面・画像に表示する店舗名
- `login_code`: 店舗ログイン用コード
- `pin_hash`: PINのハッシュ値
- `logo_url`: 店舗ロゴ画像URL
- `frame_url`: 写真枠画像URL
- `theme_color`: 店舗テーマ色
- `print_template_type`: 印刷テンプレート種別
- `is_active`: 有効/無効
- `created_at`: 作成日時

### staff_members

店舗ごとの担当者候補。

- `id`: UUID
- `store_id`: storesへの参照
- `staff_code`: 担当者コード
- `display_name`: 表示名
- `is_active`: 有効/無効
- `sort_order`: 表示順
- `created_at`: 作成日時

### assets

最終合意後に保存する完成品素材。

- `id`: UUID
- `manage_code`: 店舗コード・日付・連番から作る管理コード
- `store_id`: storesへの参照
- `store_code`: 検索・表示用の店舗コード
- `staff_id`: staff_membersへの参照
- `staff_display_name`: 保存時点の担当者表示名
- `captured_date`: 撮影日
- `sequence_number`: 店舗ごとの日次連番
- `dog_name`: わんちゃんの名前
- `dog_breed`: 犬種
- `dog_age`: 犬齢
- `sns_consent`: SNS掲載の最終合意
- `mosaic_required`: モザイク要否
- `final_processed_url`: 完成画像URL
- `status`: `ready` / `archived`
- `created_at`: 登録日時

将来的には、季節限定フレームやイベント用フレームを扱うために `store_frames` を別テーブルに分けてもよい。ただし、最初は `stores.frame_url` だけで十分。

## 7. フェーズ1の画面フロー案

```text
1. 店舗ログイン
   店舗コード + PIN

2. 撮影画面
   店舗名、日付、店舗フレームを表示
   撮影画面上で担当者を選択
   3〜4枚撮影

3. 写真選択
   お客様と一緒にベストショットを1枚選ぶ

4. 犬情報入力
   名前、犬種、犬齢を入力

5. 画像加工
   店舗ロゴ、店舗名、写真枠、手動モザイクをCanvas合成

6. 完成画像
   印刷、最終合意確認へ進む

7. 最終合意OK後だけ保存
   Supabase Storageへ完成画像を保存
   Supabase Databaseへメタデータを保存
```

## 8. クラウド保存ポリシー

クラウド保存は、印刷した写真をお客様に見せ、SNS掲載の最終合意OKをもらえた段階で初めて行う。

お客様からOKがもらえなかった場合、途中離脱した場合、キャンセルされた場合、やり直しされた場合は、クラウドへ1ミリもデータを送らない。

## 9. 禁止事項

- ローカルWindows環境での開発・ビルド・依存インストールを前提にしない。
- APIキー、DB URL、Storage URL、店舗名、環境固有URLをコードへ直書きしない。
- 店舗ごとのフレームや店名をハードコードしない。
- 未許諾写真、ボツ写真、途中データをSupabaseへ送らない。
- Meta API連携、SNS自動投稿機能、本部管理画面など、まだ合意していない機能を先走って実装しない。
- 印刷はWeb標準の `window.print()` と CSS `@media print` を基本とし、高度な印刷制御ライブラリへ安易に依存しない。
