# 店舗分散型SNS自動投稿システム 設計・仕様

最終更新: 2026-07-13

この文書は、画面仕様、DB設計、保存方針、現在の開発方針を引き継ぐための基本設計書です。

プロジェクト全体の判断原則は `docs/project-principles.md` を最上位とします。新しいClaude Codeスレッドの開始手順は `docs/START_HERE.md`、読む文書の分類は `docs/DOC_INDEX.md` を参照します。

## 1. プロジェクトの目的

このプロジェクトは、ユーザー本人の長期事業ではなく、友人が経営する飲食店を支援するためのものです。

東京・軽井沢に複数店舗を展開する、わんちゃん同伴OKのイタリアンレストラン向けに、「今日のわんちゃん」写真を店頭で撮影、加工し（希望があれば印刷し）、SNS掲載OKをもらった完成画像だけをクラウドへ保存する仕組みを作ります。

成功条件は、飲食店の忙しい現場でも迷わず使えることです。ITに詳しくないスタッフでも、iPadだけで担当者選択、撮影、写真選定、画像加工、印刷（任意）、最終合意、保存まで完了できる体験を優先します。

## 2. 全体アーキテクチャ

システムは3つのレイヤーに分けます。

1. 店舗側iPad Webアプリ
   - 店頭で担当者選択、撮影、写真選択、画像編集、印刷（任意）、同意後保存を行う。
   - 現在の主対象。

2. 本部側 `/admin`
   - 保存済み完成画像、店舗、担当者、写真枠を管理する。
   - 将来的に投稿素材整理や投稿準備を補助する可能性がある。

3. SNS投稿
   - 完成画像と文章を、確認済みの運用手順に沿ってInstagram等へ投稿する。
   - 後続フェーズの対象。

各レイヤーは疎結合にします。撮影、画像加工、DB保存、本部確認、投稿文作成、SNS投稿を密結合させません。

## 3. 技術スタック

- フロントエンド: Next.js
- ホスティング: Vercel
- DB: Supabase Database
- Storage: Supabase Storage
- Auth: Supabase Auth
- 画像加工: ブラウザ内の Canvas API
- 顔検出: 使わない。スタッフによる手動モザイクのみ（未使用だった顔検出ライブラリ一式は2026-07-13に削除済み）
- 印刷: Web標準の `window.print()` と CSS `@media print`
- 対象端末: iPad Safari
- 対象プリンター: Canon SELPHY、AirPrint

接続情報、APIキー、環境固有URLはコードに直書きしません。VercelとSupabase側の設定で管理します。

## 4. 現在できていること

Vercel上のプロトタイプで、以下の流れが動作しています。

1. 店舗コード + PINで店舗ログインする
2. Supabaseから店舗情報、担当者一覧、写真枠を取得する
3. 店舗ごとに最大3種類の有効な枠を撮影画面で選択できる
4. iPad Safariでカメラを起動する
5. 撮影画面で担当者を選択する（担当者を選ぶまで撮影ボタンは押せない）
6. 最大3枚まで撮影する
7. 撮影直後の画像は長辺2400px、JPEG品質0.9を上限に軽量化する
8. 撮影画像はクラウドへ送らず、ブラウザメモリ内のBlob/Object URLだけで一時保持する
9. 3枚から1枚を選ぶ
10. 選択済みの担当者情報は画像編集、完成画像、保存データへ引き継がれる
11. 画像編集画面で写真だけをタッチで移動、拡大縮小、回転する
12. 必要な場所だけ手動モザイクする
13. 直前のモザイクだけ取り消す
14. 写真上へ短い文字ボックスを追加、選択、直接編集、水平化、削除できる
15. 文字ボックスは写真に連動して移動、拡大縮小、回転する
16. 正方形の完成画像を生成してプレビューする
17. 完成画像確認画面から編集へ戻れる
18. 印刷ボタンでブラウザ標準の印刷画面を開く
19. 印刷時は完成画像だけを印刷対象にする
20. SNS掲載OKをもらった正方形完成画像だけをSupabase StorageとDBへ保存できる
21. 本部用 `/admin` で保存済み写真を一覧確認できる
22. 本部用 `/admin` で写真一覧の複数選択と、詳細画面での順番確認ができる
23. 本部用 `/admin` で選択した写真の一言メモと確認状態を追加、更新できる
24. 本部用 `/admin` で選択した写真を削除せず非表示にし、必要なら表示へ戻せる
25. 本部用 `/admin` で店舗マスタ、担当者マスタ、店舗ごとの写真枠を管理できる

完成画像保存を使うには、Supabase側で現在の `schema.sql` または必要なmigrationを適用している必要があります。
本部用 `/admin` を使うには、Vercelに `ADMIN_MAINTENANCE_PIN` を設定する必要があります。

## 5. 店舗側画面フロー

```text
1. 店舗ログイン
   店舗コード + PIN

2. 撮影画面
   店舗名、日付、店舗フレームを表示
   撮影する担当者を選ぶ（選ぶまで撮影ボタンは押せない）
   最大3枚まで撮影
   撮影画像は長辺2400pxまでに縮小して一時保持

3. 写真選択
   お客様と一緒にベストショットを1枚選ぶ

4. 画像編集
   写真の移動、拡大縮小、回転
   手動モザイク
   写真上への短い文字追加
   店舗名、写真枠、日付を正方形Canvasへ合成
   ロゴは写真枠のデザイン内に含める

5. 完成画像確認
   正方形の完成画像を確認
   必要なら編集へ戻る
   お客様が希望した場合だけ印刷ボタンで印刷（印刷は任意）

6. 最終合意後だけ保存
   お客様からSNS掲載OKをもらった場合だけ保存
   Supabase Storageへ正方形完成画像を保存
   Supabase Databaseへメタデータを保存
```

## 6. 担当者選択の方針

担当者はログインではなく、撮影ごとに選ぶメタ情報として扱います。

- iPadは個人ではなく店舗としてログインする
- 当日、同じiPadを複数名で使う可能性がある
- 担当者は撮影前に選択する。選ぶまで撮影ボタンは押せない（当初は写真選択後の選択だったが、担当者の選び忘れを防ぐため撮影前へ変更した）
- 担当者は大きなボタンで選ぶ
- 選択された担当者は、画像編集、完成画像表示、保存データへ引き継ぐ
- 保存時には、その時点で選ばれていた `staff_id` と `staff_display_name` を素材データへ持たせる

担当者の追加、編集はSupabase側または本部管理画面で行い、現場画面では選ぶだけにします。

## 7. わんちゃん情報と文字ボックスの方針

犬種、犬齢、オス・メス、名前などをDB項目として保存する方針は一旦やめます。

現在の方針では、わんちゃん情報はDBの構造化データではなく、写真上に焼き込む短い文字として扱います。

理由:

- 一度に1匹とは限らず、複数匹になることがある
- 現場で犬ごとの細かいフォーム入力を行うと負担が重い
- 最終目的はSNS投稿やプリントであり、写真上に分かりやすく表示できればよい
- DB上で犬情報を正規化するより、写真上の自由な文字配置の方が現場に合っている

文字ボックス仕様:

- 「文字を追加」ボタンで追加する
- 最大8個まで
- 1つの文字ボックスは15文字以内
- 既存の文字を指で触ると、その文字が選択され編集できる
- 選択中の文字は写真上に直接入力欄を重ね、その場で内容を編集できる
- 内容、文字サイズ、文字色を変更できる
- 不要な文字は削除できる
- 文字サイズは `小`、`中`、`大`
- 文字色は `白`、`黒`、`赤`、`青`、`黄`
- 文字は写真座標に紐づく
- 写真を動かす、拡大縮小する、回転すると、文字も同じように連動する
- 写真全体の回転で文字が傾いた場合は、選択中の文字だけ画面上で水平に戻せる
- 最終画像では選択枠などの編集UIは焼き込まない

DBには、文字ボックスの内容や犬情報を個別項目として保存しません。完成画像に焼き込まれた見た目を正とします。

## 8. 店舗ごとの写真枠・ロゴ

店舗ごとの表示情報はコードに書かず、Supabaseの店舗マスタと写真枠マスタから取得します。

現在の方針:

- 写真枠は正方形完成画像用の枠を標準とする
- 写真枠は店舗ごとに2〜3種類まで有効にできる
- 日付表示位置、文字サイズ、色は枠ごとの設定として持つ
- ロゴは単独素材ではなく、写真枠画像のデザイン内に含める
- 店舗画面と完成画像では、ロゴ単体画像を重ねない
- 既存の `stores.logo_url` やロゴアップロード機能は、互換性のため一時的に残す
- 枠画像への移行が済んだら、ロゴ管理は削除または非表示にする
- 既存のL版前提の枠は、正方形化後は旧形式または停止中の素材として扱う

店舗画面・完成画像で使う主な要素:

- 撮影画面の半透明フレーム
- Canvas合成時の店舗名
- Canvas合成時の写真枠
- 枠ごとの日付表示設定
- 印刷用テンプレート
- 表示カラー

これらは、ログイン中の店舗設定から動的に反映します。枠が未設定の場合は、画面が壊れないように仮表示へフォールバックします。

## 9. Supabaseデータ設計

詳細なSQLは `supabase/schema.sql` と `supabase/migrations/` を正とします。この章は設計意図の要約です。

### stores

店舗マスタ。

主な項目:

- `id`: UUID
- `store_code`: 店舗コード
- `store_name`: 正式店舗名
- `display_name`: 画面・画像に表示する店舗名
- `login_code`: 店舗ログイン用コード
- `pin_hash`: PINのハッシュ値
- `theme_color`: 店舗テーマ色
- `timezone`: 店舗のタイムゾーン
- `is_active`: 有効/無効
- `sort_order`: 管理画面での表示順
- `notes`: 管理用メモ

互換用または後続整理予定の項目:

- `logo_url`
- `frame_url`
- `sns_display_name`
- `instagram_account`
- `default_hashtags`
- `address`
- `phone`
- `business_hours_note`

店舗が閉店・休止した場合も物理削除せず、原則として `is_active = false` にします。

### staff_members

店舗ごとの担当者候補。

主な項目:

- `id`: UUID
- `store_id`: storesへの参照
- `staff_code`: 内部管理用コード（本部管理画面では入力せず、サーバー側で自動生成する）
- `display_name`: 担当者名（本部管理画面で編集する唯一の名前項目）
- `is_active`: 有効/無効
- `sort_order`: 表示順
- `notes`: 管理用メモ

`role_label`（役割ラベル）と `can_approve_sns`（SNS掲載確認を担当できるか）は2026-07-09に廃止しました。店舗側の撮影・保存処理では一切参照されておらず、実質的に何も制御していなかったためです。

担当者が退職・異動した場合も物理削除せず、原則として `is_active = false` にします。過去の素材データには保存時点の `staff_display_name` も残します。

### assets

最終合意後に保存する完成品素材。

主な項目:

- `id`: UUID
- `manage_code`: 店舗コード・日付・連番から作る管理コード
- `store_id`: storesへの参照
- `store_code`: 検索・表示用の店舗コード
- `store_display_name`: 保存時点の店舗表示名
- `staff_id`: staff_membersへの参照
- `staff_display_name`: 保存時点の担当者表示名
- `captured_at`: 撮影日時
- `captured_date`: 撮影日
- `sequence_number`: 店舗ごとの日次連番
- `short_caption`: 本部または店舗が使う40文字以内の任意一言メモ
- `review_status`: 本部確認状態
- `sns_consent`: SNS掲載の最終合意
- `mosaic_required`: モザイク要否
- `final_processed_url`: 完成画像URL
- `final_storage_bucket`: 完成画像を保存したStorage bucket
- `final_storage_path`: 完成画像を保存したStorage path
- `theme_color_snapshot`: 保存時点のテーマカラー
- `printed_at`: 印刷した日時
- `consent_confirmed_at`: SNS掲載合意を確認した日時
- `hidden_at`: 本部が非表示にした日時
- `hidden_reason`: 非表示理由
- `status`: `ready` / `archived`

`description`、`logo_url_snapshot`、`frame_url_snapshot` は旧来または互換用の項目として扱います。完成画像には枠が合成済みなので、枠IDや枠URLは必須の保存メタ情報ではありません。

犬情報項目は新規設計では使いません。完成画像に焼き込まれた見た目を正とします。

### store_frames

店舗別写真枠テーブル。

主な項目:

- `id`: UUID
- `store_id`: storesへの参照
- `frame_name`: フレーム名（本部管理画面では入力せず、サーバー側で自動生成する内部識別用）
- `frame_url`: フレーム画像URL
- `is_default`: 標準フレームか（サムネイル3枠のうち一番左が標準）
- `starts_at`: 利用開始日時（現状未使用）
- `ends_at`: 利用終了日時（現状未使用）
- `sort_order`: 表示順
- `date_enabled`: 枠の日付表示を使うか
- `date_x`: 日付のX座標
- `date_y`: 日付のY座標
- `date_font_size`: 日付の文字サイズ
- `date_color`: 日付の文字色

店舗ごとに最大3枠まで（`MAX_FRAMES_PER_STORE`）。`is_active` による無効化は2026-07-08に廃止し、削除は直接のレコード削除（ハードデリート）にしました。枠は写真に合成済みで保存されるため、枠自体を後から履歴参照する必要がないためです。

## 10. クラウド保存ポリシー

クラウド保存は、完成画像をお客様に見せ、SNS掲載の最終合意OKをもらえた段階で初めて行います。印刷は保存の前提条件ではなく、お客様が希望した場合だけ行う任意の出力です。

保存するもの:

- 編集後の写真、文字、モザイク、正方形枠を合成した正方形完成画像
- 店舗ID、店舗コード、店舗表示名
- 担当者ID、担当者表示名
- 撮影日時、保存日時
- 印刷日時（印刷した場合のみ。現状は印刷ボタンを押した時点の時刻で、印刷ダイアログのキャンセルは区別しない）
- SNS掲載合意日時
- 任意の一言メモ
- 管理ステータス

保存しないもの:

- 撮影途中の写真
- 選ばれなかった写真
- 編集途中のCanvas状態
- 未許諾写真
- キャンセルされた写真
- 犬種、犬齢、性別、名前などの構造化された犬情報
- 枠IDや枠URLを必須の保存メタ情報として扱うこと

お客様からOKがもらえなかった場合、途中離脱した場合、キャンセルされた場合、やり直しされた場合は、クラウドへ画像を送りません。

## 11. 印刷方針

印刷は、完成画像確認画面の印刷ボタンから行います。印刷は任意で、保存の前提条件ではありません（印刷せずに保存できます）。

保存素材は正方形完成画像を正とします。印刷は店舗でお客様に確認してもらうための出力手段として扱い、用紙サイズや向きは実機確認の結果に合わせて調整します。

- ブラウザ標準の印刷機能を使う
- 印刷時だけ、ボタンや画面上の補助UIを非表示にする
- 完成画像だけを印刷対象にする
- Canon SELPHY、AirPrint、iPad Safariで実機確認する

注意:

- AirPrint側やSELPHY側の用紙設定、フチなし設定に左右される
- 最初は1枚だけテスト印刷する
- 画像切れ、余白、向き、色味は実機確認後に調整する

## 12. 本部メンテナンス画面 `/admin`

現時点の `/admin` は、保存済み写真の確認、店舗マスタ・担当者マスタ、写真枠の軽いメンテナンスに絞ります。

できること:

- 本部PINで開く
- タブで `写真`、`店舗`、`枠` を切り替える（担当者管理は2026-07-09に「店舗」タブへ統合した）
- 保存済み写真を一覧表示する
- 店舗、日付範囲、確認状態で絞り込む
- 非表示写真を含めて表示するか切り替える
- 日付順、店順で写真を並べ替える
- 日付順は新しい順、古い順を切り替える
- 写真一覧でチェックボックスによる複数選択を行う
- 選択した写真を詳細画面で順番に確認する
- 写真の一言メモと確認状態を追加、更新する
- 写真を削除せず `archived` として非表示にする
- 非表示にした写真を `ready` に戻す
- 写真を完全に削除する（確認ダイアログの後、Storage画像とDBレコードの両方を削除。SNS掲載同意後でも顧客からの削除申し入れに対応できるよう、常時使える操作として用意している）
- 店舗の店舗名、表示名、並び順、有効/無効、メモを編集する。編集パネル内に、その店舗の担当者一覧・追加・編集も表示する
- 担当者の追加、担当者名、有効/無効、並び順、メモを編集する（役割・SNS承認可の属性は2026-07-09に廃止済み）
- 店舗ごとの写真枠（最大3枚）をサムネイルで追加、差し替え、並び替え、削除する。一番左が標準枠

まだ作らないこと:

- SNS投稿文の作成と保存
- SNS自動投稿
- 店舗ログインコードとPINハッシュの画面編集
- 複雑な管理者権限
- 複数写真をまとめて同じ値へ更新する一括更新
- 店舗側からの写真削除（削除は本部側のみ）

今後削除または非表示にすること:

- ロゴURLの編集
- ロゴ画像そのもののアップロード
- ロゴ単独素材を前提にした保存項目

写真の完全削除は事故の影響が大きいため、日常運用ではまず非表示（`archived`）を優先し、完全削除は顧客からの申し入れなど必要な場合だけ使う運用とします。枠については、写真に合成済みで保存されるため履歴参照の必要がなく、完全削除を通常運用にしています。

## 13. 当面の優先事項

現在は、新機能を増やすよりも、店舗側と本部側の基本動作、正方形枠、保存方針、ドキュメント構成を安定させます。

当面の優先順:

1. 店舗側の撮影、編集、印刷、同意後保存の流れをiPad実機で確認する
2. 正方形枠と正方形完成画像の見え方、保存、管理画面表示を安定させる
3. `/admin` 写真タブ、店舗タブ（担当者管理を含む）、枠タブの安定確認
4. 残っているロゴ管理を削除または非表示にするか判断
5. 枠と基本保存の流れが安定してから、SNS投稿素材や外部連携の扱いを改めて検討する

## 14. 先走らないこと

- 目的、OK条件、確認方法が曖昧なまま実装しない
- APIキー、DB URL、Storage URL、環境固有URLをコードへ直書きしない
- 店舗ごとのフレームや店名をハードコードしない
- 未許諾写真、ボツ写真、途中データをSupabaseへ送らない
- 犬情報をDB項目として先走って固定しない
- ローカルPCやDropboxにしかない素材や設定を正本として扱わない
- SNS投稿、自動投稿、AI連携、権限管理を先走って実装しない
- Meta API連携、SNS自動投稿機能、本部管理画面の高度な機能など、まだ合意していない機能を先走って実装しない
- 印刷はWeb標準の `window.print()` と CSS `@media print` を基本とし、高度な印刷制御ライブラリへ安易に依存しない
