# 店舗分散型SNS自動投稿システム 設計・開発方針

このファイルは、新しいスレッドや別作業者へ開発を引き継ぐための最重要ドキュメントです。開発前に必ずこのファイルを読み、ここに書かれた方針を最優先すること。

## 0. 最重要方針: ローカルに何も持たない

このプロジェクトでは、ユーザーのWindowsローカル環境にソースコード、依存関係、ビルド環境、実行環境を持たない方針を最優先とする。

- ソースコードはGitHubに集約する
- 実行・ビルド・プレビューはVercelで行う
- DB、Storage、Auth、店舗設定、完成画像はSupabaseに集約する
- ローカルPCでは `npm install`、`npm run dev`、Next.js初期化、依存関係インストールを原則行わない
- Codexは必要な変更をGitHubへ直接Pushする
- ユーザーはVercel上のURLをiPad Safariで確認する
- ローカルに一時的な開発ファイルや複製リポジトリを作ることも避ける

ローカル環境で詰まった場合に回避するのではなく、最初からクラウド直結の開発フローを正とする。

## 1. プロジェクトの目的

東京・軽井沢に複数店舗を展開する、わんちゃん同伴OKのイタリアンレストラン向けに、「今日のわんちゃん」写真を店頭で撮影・加工・保存し、将来的にSNSへ自動投稿できる仕組みを作る。

最終目標は、現場スタッフの負担を増やさず、愛犬家コミュニティへのブランド露出を継続的に高めること。

成功条件は、飲食店の忙しい現場でも迷わず使えること。ITに詳しくないスタッフでも、iPadだけで撮影、写真選定、画像加工、プリント、最終合意、保存まで完了できる体験を優先する。

## 2. 全体アーキテクチャ

システムは3つのレイヤーに分ける。

1. インプット: 現場のiPad Webアプリ
   店頭で複数枚撮影し、お客様と1枚を選び、日付・店名・必要なモザイクを合成して、記念プリントとクラウド保存を行う。フェーズ1の主対象。

2. プロセス: 本部の管理画面
   各店舗から上がってきた加工済み素材を確認し、投稿対象を選ぶ。将来的にGemini APIなどで投稿文を生成する。

3. アウトプット: ManusなどによるSNS投稿
   完成した画像と文章を、指示書通りにInstagram等へ投稿する。後続フェーズの対象。

各レイヤーは疎結合にする。フロント画面、画像加工、DB通信、AI文章生成、Manus連携、SNS投稿を密結合させない。

## 3. 確定技術スタック

- フロントエンド: Next.js
- ホスティング: Vercel
- DB: Supabase Database
- Storage: Supabase Storage
- Auth: Supabase Auth
- 画像加工: ブラウザ内の Canvas API
- 顔検出・モザイク下ごしらえ: 自動顔判定は原則使わず、スタッフによる手動モザイクを優先する
- 印刷: Web標準の `window.print()` と CSS `@media print`
- 対象端末: iPad Safari
- 対象プリンター: Canon SELPHY、AirPrint、写真L版、フチなし想定

すべての接続情報は初日から環境変数で読む。DB URL、Storage URL、APIキー、Supabase anon key、将来のAI APIキーなどをコードに直書きしてはいけない。

## 4. フェーズ0

外部サーバーやDB通信は後回しにする。まずクラウド上のVercel環境で、カメラ撮影、3枚一時保持、選択、入力、Canvas加工、手動モザイク、完成画像化の挙動を確認する。

フェーズ0ではSupabaseへ保存しない。ブラウザメモリ内のBlob/Object URLだけを使い、撮影データの保持と破棄を確認する。

## 5. フェーズ1

1. iPad Safariで3〜4枚を連続撮影する
2. お客様と一緒にベストショットを1枚選ぶ
3. わんちゃん情報を入力する
4. Canvas APIで日付、店名、店舗フレーム、手動モザイクを合成する
5. Canon SELPHYへAirPrintでプリントする
6. プリント後にお客様へSNS掲載の最終合意を確認する
7. OKをもらえた場合だけ、Supabase DB/Storageへ完成品画像と追加情報を一括保存する
8. OKでない場合や途中離脱時は、クラウドへ一切送らずブラウザメモリから完全破棄する

## 6. iPad撮影・選定UX

iPad標準Safariのカメラ機能をHTML5/JavaScriptから使う。可能であれば `getUserMedia` によるアプリ内カメラ体験を優先する。iPad Safariの制約が強い場合は、HTMLの `input type="file" accept="image/*" capture` も検証対象にする。

撮影画面では、日付・店名・店舗固有のデザイン枠を半透明でオーバーレイ表示する。スタッフは完成イメージを見ながら3〜4枚を連続撮影する。

撮影した複数枚はクラウドに送らない。iPadブラウザのメモリ内にBlobデータまたはObject URLとして一時保持する。

## 7. クラウド保存ポリシー

クラウド保存は、印刷した写真をお客様に見せ、SNS掲載の最終合意OKをもらえた段階で初めて行う。

お客様からOKがもらえなかった場合、途中離脱した場合、キャンセルされた場合、やり直しされた場合は、クラウドへ1ミリもデータを送らない。

## 8. 禁止事項

- ローカルWindows環境での開発・ビルド・依存インストールを前提にしない。
- APIキー、DB URL、Storage URL、店舗名、環境固有URLをコードへ直書きしない。
- 店舗ごとのフレームや店名をハードコードしない。
- 未許諾写真、ボツ写真、途中データをSupabaseへ送らない。
- 合意していないMeta API連携、SNS投稿機能、独自の本部画面を先走って作らない。
- 高度な印刷制御ライブラリに依存しない。
