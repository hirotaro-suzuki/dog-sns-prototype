# 店舗分散型SNS自動投稿システム 設計・開発方針

このファイルは、新しいスレッドや別作業者へ開発を引き継ぐための最重要ドキュメントです。開発前に必ずこのファイルを読み、ここに書かれた方針を最優先すること。

最終更新: 2026-06-21

## 0. 最重要方針: GitHub mainを正とする

このプロジェクトでは、開発・確認・運用の基準をクラウド側に置く。

- ソースコードはGitHubに集約する
- GitHubの `main` ブランチを最新版・正本として扱う
- 実行・ビルド・プレビューはVercelで行う
- DB、Storage、Auth、店舗設定、完成画像はSupabaseに集約する
- ローカルPC上の作業環境や複製リポジトリを前提にしない
- Codexは必要な変更をGitHubへ直接反映する
- ユーザーはVercel上のURLをiPad Safariで確認する

困った時にローカルへ退避するのではなく、GitHub、Vercel、Supabaseを正として開発を進める。

## 0.1 引き継ぎやすさを最優先する

このプロジェクトでクラウド中心の構成を採る最大の理由は、開発や運用の引き継ぎを軽くするためである。

ユーザー本人が専門的な開発環境を管理しなくても、次のCodexスレッド、別の作業者、将来の保守担当者が、GitHub、Vercel、Supabaseを見れば現在地を理解し、続きから作業できる状態を保つ。

そのため、機能を追加する時は、便利そうかどうかだけでなく「次の人が迷わず引き継げるか」を判断基準にする。

優先すること:

- GitHub上のREADMEと設計書を読めば、現在の方針と次の作業が分かる状態にする
- 設定、DB、Storage、画面フローをクラウド上で確認できるようにする
- 店舗別の情報や運用設定は、コードではなくSupabase側に寄せる
- 作業の区切りごとに、変更内容、未確認事項、次に進む候補を記録する
- 初めて見る人でも追えるように、画面フロー、保存条件、禁止事項を文書に残す

避けること:

- 特定のPC、特定のフォルダ、特定の人の手元環境がないと再開できない構成
- 口頭説明や記憶に頼らないと分からない仕様
- 一時的な都合で、本番方針と違う場所に重要な情報を置くこと
- 引き継ぎに手間がかかる複雑な構成や、過度に属人的な運用

迷った場合は、短期的な作りやすさよりも、後から引き継ぎやすい構成を優先する。

## 0.2 開発判断はCodexを中心にする

以前は、ターミナル画面や作業結果を別のAIへ渡して判断を確認する運用も想定していたが、現在はCodexを中心に進める。

Codexは、GitHub上の設計書、コード、Supabase関連ファイル、Vercel前提の開発方針を読み、実装、確認、記録まで一貫して担当する。別AIへ都度判断を渡す前提にはしない。

将来、SNS投稿文の自動生成などでAI機能が必要になった場合は、その時点で用途、入力データ、保存先、確認フローを整理してから、利用するAIや連携方法を改めて検討する。

## 1. プロジェクトの目的

東京・軽井沢に複数店舗を展開する、わんちゃん同伴OKのイタリアンレストラン向けに、「今日のわんちゃん」写真を店頭で撮影・加工・保存し、将来的にSNSへ投稿できる仕組みを作る。

最終目標は、現場スタッフの負担を増やさず、愛犬家コミュニティへのブランド露出を継続的に高めること。

成功条件は、飲食店の忙しい現場でも迷わず使えること。ITに詳しくないスタッフでも、iPadだけで撮影、写真選定、担当者選択、画像加工、プリント、最終合意、保存まで完了できる体験を優先する。

## 2. 全体アーキテクチャ

システムは3つのレイヤーに分ける。

1. インプット: 現場のiPad Webアプリ
   店頭で複数枚撮影し、お客様と1枚を選び、日付・店名・写真枠・必要なモザイク・短い文字を合成して、記念プリントとクラウド保存を行う。現在の主対象。

2. プロセス: 本部の管理画面
   各店舗から上がってきた加工済み素材を確認し、投稿対象を選ぶ。店舗、担当者、完成画像の簡単なメンテナンスを行う。将来的に投稿文の作成や投稿準備を補助する機能を検討する。

3. アウトプット: SNS投稿
   完成した画像と文章を、確認済みの手順に沿ってInstagram等へ投稿する。後続フェーズの対象。

各レイヤーは疎結合にする。フロント画面、画像加工、DB通信、投稿文作成、SNS投稿を密結合させない。

## 3. 確定技術スタック

- フロントエンド: Next.js
- ホスティング: Vercel
- DB: Supabase Database
- Storage: Supabase Storage
- Auth: Supabase Auth
- 画像加工: ブラウザ内の Canvas API
- 顔検出・モザイク下ごしらえ: 自動顔判定は原則使わず、スタッフによる手動モザイクを優先する
- 印刷: Web標準の `window.print()` と CSS `@media print`
- 対象端末: iPad Safari
- 対象プリンター: Canon SELPHY、AirPrint、写真L版、フチなし想定

すべての接続情報は初日から環境変数で読む。DB URL、Storage URL、APIキー、Supabase anon key、将来必要になる外部連携キーなどをコードに直書きしてはいけない。

## 4. 現在できていること

Vercel上のプロトタイプで、以下の流れが動作している。

1. 店舗コード + PINで店舗ログインする
2. Supabaseから店舗情報と担当者一覧を取得する
3. iPad Safariでカメラを起動する
4. 最大3枚まで撮影する
5. 撮影直後の画像は長辺2400px、JPEG品質0.9を上限に軽量化する
6. 撮影画像はクラウドへ送らず、ブラウザメモリ内のBlob/Object URLだけで一時保持する
7. 3枚から1枚を選ぶ
8. 写真選択後に担当者を選択する
9. 担当者を選ばないと画像編集へ進めない
10. 画像編集画面で写真だけをタッチで移動、拡大縮小、回転する
11. 必要な場所だけ手動モザイクする
12. 直前のモザイクだけ取り消す
13. 写真上へ短い文字ボックスを自由に追加できる
14. 文字ボックスを指で選択し、写真上で内容、色、サイズ、水平化を編集できる
15. 文字ボックスは写真に連動して移動、拡大縮小、回転する
16. 完成画像を生成してプレビューする
17. 完成画像確認画面から編集へ戻っても、写真サイズが崩れない
18. 印刷ボタンでブラウザ標準の印刷画面を開く
19. 印刷時は完成画像だけをL版横向き想定で出力する
20. 完成画像確認画面から、SNS掲載OKをもらった完成画像をSupabase StorageとDBへ保存できる
21. 本部用 `/admin` で保存済み写真を一覧確認できる
22. 本部用 `/admin` で店舗の複数指定、日付範囲指定、非表示写真の表示切替ができる
23. 本部用 `/admin` で選択した写真の説明文を追加・更新できる
24. 本部用 `/admin` で選択した写真を削除せず非表示にし、必要なら表示へ戻せる
25. 本部用 `/admin` で店舗マスタの表示名、SNS表示名、ロゴURL、フレームURL、テーマカラー、有効/無効を編集できる
26. 本部用 `/admin` で担当者マスタの追加、名前変更、役割、有効/無効を編集できる

完成画像保存を使うには、Supabase側で現在の `schema.sql` または `supabase/migrations/20260625_assets_storage_handoff.sql` を適用する必要がある。
本部用 `/admin` を使うには、Vercelに `ADMIN_MAINTENANCE_PIN` を設定する必要がある。

## 5. 現在の画面フロー

```text
1. 店舗ログイン
   店舗コード + PIN

2. 撮影画面
   店舗名、日付、店舗フレームを表示
   最大3枚まで撮影
   撮影画像は長辺2400pxまでに縮小して一時保持

3. 写真選択
   お客様と一緒にベストショットを1枚選ぶ

4. 担当者選択
   写真選択後、その撮影を担当したスタッフを選ぶ
   担当者を選ばないと編集へ進めない

5. 画像編集
   写真の移動、拡大縮小、回転
   手動モザイク
   写真上への短い文字追加
   店舗ロゴ、店舗名、写真枠、日付を合成

6. 完成画像確認
   完成画像を確認
   必要なら編集へ戻る
   印刷ボタンで印刷

7. 最終合意後だけ保存
   印刷後、お客様からSNS掲載OKをもらった場合だけ保存
   Supabase Storageへ完成画像を保存
   Supabase Databaseへメタデータを保存
```

## 6. 担当者選択の方針

担当者はログインではなく、撮影ごとに選ぶメタ情報として扱う。

- iPadは個人ではなく「店舗」としてログインする
- 当日、同じiPadを複数名で使う可能性がある
- 担当者は撮影前ではなく、写真を1枚選んだ後に選択する
- 担当者は大きなボタンで選ぶ
- 選択された担当者は、画像編集、完成画像表示、保存データへ引き継ぐ
- 保存時には、その時点で選ばれていた `staff_id` と `staff_display_name` を素材データへ持たせる

フェーズ1では、担当者の追加・編集はSupabase側または本部管理画面で行い、現場画面では選ぶだけにする。現場で自由追加できる機能は、必要性が見えてから検討する。

## 7. わんちゃん情報と文字ボックスの方針

当初検討していた、犬種、犬齢、オス・メス、名前などをDB項目として保存する方針は一旦やめる。

現在の方針では、わんちゃん情報はDBの構造化データではなく、写真上に焼き込む短い文字として扱う。

理由:

- 一度に1匹とは限らず、複数匹になることがある
- 現場で犬ごとの細かいフォーム入力を行うと負担が重い
- 最終目的はSNS投稿やプリントであり、写真上に分かりやすく表示できればよい
- DB上で犬情報を正規化するより、写真上の自由な文字配置の方が現場に合っている

文字ボックス仕様:

- 「文字を追加」ボタンで追加する
- 最大8個まで
- 1つの文字ボックスは15文字以内
- 既存の文字を指で触ると、その文字が選択され編集できる
- 選択中の文字は写真上に直接入力欄を重ね、その場で内容を編集できる
- 選択中の文字の近くに、水平、文字サイズ、文字色、削除の操作をまとめて表示する
- 内容、文字サイズ、文字色を変更できる
- 不要な文字は削除できる
- 文字サイズは `小`、`中`、`大`
- 文字色は `白`、`黒`、`赤`、`青`、`黄`
- 文字は写真座標に紐づく
- 写真を動かす、拡大縮小する、回転すると、文字も同じように連動する
- 写真全体の回転で文字が傾いた場合は、選択中の文字だけ画面上で水平に戻せる
- 最終画像では選択枠などの編集UIは焼き込まない

DBには、文字ボックスの内容や犬情報を個別項目として保存しない。完成画像に焼き込まれた見た目を正とする。

## 8. 店舗ごとの写真枠・ロゴ

店舗ごとの表示情報はコードに書かず、Supabaseの店舗マスタから取得する。

- 撮影画面の半透明フレーム
- Canvas合成時の店舗名
- Canvas合成時の店舗ロゴ
- 印刷用テンプレート
- 表示カラー

これらは、ログイン中の店舗設定から動的に反映する。ロゴやフレームが未設定の場合は、画面が壊れないように仮表示へフォールバックする。

## 9. Supabaseデータ設計案

最初に必要なテーブルは以下。

### stores

店舗マスタ。

- `id`: UUID
- `store_code`: 店舗コード
- `store_name`: 正式店舗名
- `display_name`: 画面・画像に表示する店舗名
- `login_code`: 店舗ログイン用コード
- `pin_hash`: PINのハッシュ値
- `logo_url`: 店舗ロゴ画像URL
- `frame_url`: 写真枠画像URL
- `theme_color`: 店舗テーマ色
- `print_template_type`: 印刷テンプレート種別
- `timezone`: 店舗のタイムゾーン
- `sns_display_name`: SNS投稿時の表示名
- `instagram_account`: Instagramアカウント名
- `default_hashtags`: 標準ハッシュタグ
- `address`: 店舗住所
- `phone`: 店舗電話番号
- `business_hours_note`: 営業時間メモ
- `is_active`: 有効/無効
- `sort_order`: 管理画面での表示順
- `notes`: 管理用メモ
- `created_at`: 作成日時
- `updated_at`: 更新日時

店舗が閉店・休止した場合も物理削除せず、原則として `is_active = false` にする。過去の素材データとのつながりを壊さないためである。

### staff_members

店舗ごとの担当者候補。

- `id`: UUID
- `store_id`: storesへの参照
- `staff_code`: 担当者コード
- `display_name`: 表示名
- `role_label`: 役割ラベル
- `can_approve_sns`: SNS掲載確認を担当できるか
- `is_active`: 有効/無効
- `sort_order`: 表示順
- `notes`: 管理用メモ
- `created_at`: 作成日時
- `updated_at`: 更新日時

担当者が退職・異動した場合も物理削除せず、原則として `is_active = false` にする。過去の素材データには保存時点の `staff_display_name` も残す。

### assets

最終合意後に保存する完成品素材。

保存する中心情報:

- `id`: UUID
- `manage_code`: 店舗コード・日付・連番から作る管理コード
- `store_id`: storesへの参照
- `store_code`: 検索・表示用の店舗コード
- `store_display_name`: 保存時点の店舗表示名
- `staff_id`: staff_membersへの参照
- `staff_display_name`: 保存時点の担当者表示名
- `captured_at`: 撮影日時
- `captured_date`: 撮影日
- `sequence_number`: 店舗ごとの日次連番
- `description`: 本部が後から追加・更新する写真説明文
- `sns_consent`: SNS掲載の最終合意
- `mosaic_required`: モザイク要否
- `final_processed_url`: 完成画像URL
- `final_storage_bucket`: 完成画像を保存したStorage bucket
- `final_storage_path`: 完成画像を保存したStorage path
- `frame_url_snapshot`: 保存時点の写真枠URL
- `logo_url_snapshot`: 保存時点のロゴURL
- `theme_color_snapshot`: 保存時点のテーマカラー
- `print_template_type_snapshot`: 保存時点の印刷テンプレート種別
- `printed_at`: 印刷した日時
- `consent_confirmed_at`: SNS掲載合意を確認した日時
- `hidden_at`: 本部が非表示にした日時
- `hidden_reason`: 非表示理由
- `status`: `ready` / `archived`
- `created_at`: 登録日時
- `updated_at`: 更新日時

現時点では、以下の犬情報項目は新規設計では使わない。

- `dog_name`
- `dog_breed`
- `dog_age`
- `dog_gender`

既存テーブルや既存SQLにこれらの項目が残っている場合は、次のDB保存実装前に整理する。削除するか、互換性のため任意項目として残すかは、実際のSupabase状態を確認してから決める。

完成画像は、印刷後にお客様からSNS掲載OKをもらえた場合だけ保存する。未許諾写真、ボツ写真、途中データは保存しない。

完成画像本体はSupabase Storageの `final-images` bucketへ保存する。店舗ロゴや写真フレームは `store-assets` bucketへ保存し、`stores.logo_url` と `stores.frame_url` に公開URLを登録する。

### admin_users

本部・管理者画面用の管理者マスタ。管理機能の作り込みは後回しでも、DB設計としては最初から用意する。

- `id`: UUID
- `auth_user_id`: Supabase AuthユーザーID
- `display_name`: 管理画面での表示名
- `role`: 管理者権限
- `is_active`: 有効/無効
- `created_at`: 作成日時
- `updated_at`: 更新日時

### store_frames

季節限定フレームやイベント用フレームを頻繁に扱う場合に追加する候補テーブル。最初は `stores.frame_url` だけでもよい。

- `id`: UUID
- `store_id`: storesへの参照
- `frame_name`: フレーム名
- `frame_url`: フレーム画像URL
- `is_default`: 標準フレームか
- `is_active`: 有効/無効
- `starts_at`: 利用開始日時
- `ends_at`: 利用終了日時
- `sort_order`: 表示順
- `created_at`: 作成日時
- `updated_at`: 更新日時

## 10. クラウド保存ポリシー

クラウド保存は、印刷した写真をお客様に見せ、SNS掲載の最終合意OKをもらえた段階で初めて行う。

保存するもの:

- 完成画像
- 店舗ID、店舗コード、店舗表示名
- 担当者ID、担当者表示名
- 撮影日時、保存日時
- 使用したロゴ、写真枠、テーマ色のスナップショット
- 印刷日時
- SNS掲載合意日時
- 管理ステータス

保存しないもの:

- 撮影途中の写真
- 選ばれなかった写真
- 編集途中のCanvas状態
- 未許諾写真
- キャンセルされた写真
- 犬種、犬齢、性別、名前などの構造化された犬情報

お客様からOKがもらえなかった場合、途中離脱した場合、キャンセルされた場合、やり直しされた場合は、クラウドへ画像を送らない。

## 11. 印刷方針

印刷は、完成画像確認画面の印刷ボタンから行う。

- ブラウザ標準の印刷機能を使う
- 印刷時だけ、ボタンや画面上の補助UIを非表示にする
- 完成画像だけを印刷対象にする
- L版横向き相当の `127mm x 89mm` を想定する
- Canon SELPHY、AirPrint、iPad Safariで実機確認する

注意:

- AirPrint側やSELPHY側の用紙設定、フチなし設定に左右される
- 最初は1枚だけテスト印刷する
- 画像切れ、余白、向き、色味は実機確認後に調整する

## 12. 本部メンテナンス画面

現時点の `/admin` は、保存済み写真の確認、店舗マスタ、担当者マスタの軽いメンテナンスに絞る。

できること:

- 本部PINで開く
- タブで `写真`、`店舗`、`担当者` を切り替える
- 保存済み写真を大きめのサムネイルで一覧する
- 店舗を複数選んで絞り込む
- 開始日・終了日で絞り込む
- 非表示写真を含めて表示するか切り替える
- 写真を選択して、管理番号、店舗、担当者を確認する
- 写真説明文を追加・更新する
- 写真を削除せず `archived` として非表示にする
- 非表示にした写真を `ready` に戻す
- 店舗の表示名、SNS表示名、Instagram、標準ハッシュタグ、ロゴURL、フレームURL、テーマカラー、有効/無効、メモを編集する
- 担当者の追加、表示名、役割、SNS承認可否、有効/無効、並び順、メモを編集する

まだ作らないこと:

- SNS投稿文の作成と保存
- SNS自動投稿
- 完全削除
- ロゴ、フレーム画像そのもののアップロード
- 店舗ログインコードとPINハッシュの画面編集
- 複雑な管理者権限

完全削除は事故の影響が大きいので、最初は非表示運用を基本にする。
SNS投稿文は次回、保存済み写真から投稿候補を選ぶ流れとして追加する。

## 13. 次フェーズの作業手順

おすすめの順序:

1. 店舗側で完成画像を保存し、`/admin` の写真一覧に出ることを確認する
2. 店舗複数指定、日付範囲指定、非表示/復帰、説明文保存をiPadで確認する
3. 店舗タブでデモ店舗名を実店舗名へ変更する
4. 担当者タブで担当者追加、有効/無効、並び順を確認する
5. 次回、選択写真からSNS投稿内容を作成・保存する機能へ進む

現場用画面と管理用画面は分ける。

```text
/store
店舗iPad用。店舗ログイン、撮影、加工、印刷、合意後保存を行う。

/admin
本部・管理者用。現時点では保存済み写真の一覧、説明文更新、非表示/復帰、店舗マスタ、担当者マスタを扱う。
```

ロゴ・フレーム画像のアップロード、店舗PIN変更、SNS投稿文作成は次以降の作業とする。

## 14. 禁止事項

- GitHub `main` 以外を最新版として扱わない。
- APIキー、DB URL、Storage URL、店舗名、環境固有URLをコードへ直書きしない。
- 店舗ごとのフレームや店名をハードコードしない。
- 未許諾写真、ボツ写真、途中データをSupabaseへ送らない。
- 犬情報をDB項目として先走って固定しない。
- 特定のPC、特定のフォルダ、特定の人の手元環境がないと再開できない構成にしない。
- 口頭説明や記憶に頼らないと分からない仕様を増やさない。
- 引き継ぎに手間のかかる複雑な構成や、過度に属人的な運用を採用しない。
- Meta API連携、SNS自動投稿機能、本部管理画面の高度な機能など、まだ合意していない機能を先走って実装しない。
- 印刷はWeb標準の `window.print()` と CSS `@media print` を基本とし、高度な印刷制御ライブラリへ安易に依存しない。
